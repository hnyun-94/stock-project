# E2E 장애 대응 작업계획서 (분업형)

작성일: 2026-03-07  
대상: `tests/test_e2e_dryrun.py` + 품질 게이트(`tests/services/`)

---

## 1) 목표

- E2E 테스트 실행 중 장애 발생 시 원인을 빠르게 분류/재현/해결한다.
- 해결 과정을 기록 가능하게 표준화한다.
- 장애가 없더라도 동일 절차로 검증해 운영 리스크를 낮춘다.

## 2) 분업 구조 (다양한 관점)

### A. QA 관점

- 테스트 실행/재현 책임
- 실패 테스트 식별 및 최소 재현 케이스 정리
- 성공 기준:
  - 품질 게이트 명령 결과 확보
  - E2E 단독 실행 결과 확보

### B. 백엔드 관점

- 실패 지점의 코드 원인 분석
- 회귀 영향도 확인(`models`, `main`, `ai_summarizer`, `utils`)
- 성공 기준:
  - 원인/영향 범위 명확화
  - 필요한 수정안 제시

### C. AI/데이터 관점

- 캐시/중복제거/프롬프트/요약 경로의 데이터 흐름 검증
- JSON 출력/파싱/폴백 경로 점검
- 성공 기준:
  - 데이터 흐름 단절 없음
  - 폴백 동작의 안정성 확인

### D. 운영/DevOps 관점

- 로컬/훅/CI 동일 품질 게이트 보장
- `pre-push` 훅 결과 및 실행 가능성 확인
- 성공 기준:
  - `core.hooksPath=.githooks`
  - 훅 실행 시 테스트 통과

## 3) 실행 순서

1. 품질 게이트 실행: `python -m pytest tests/services/ tests/test_e2e_dryrun.py -q`
2. E2E 단독 실행: `python -m pytest tests/test_e2e_dryrun.py -v`
3. 실패 시:
   - 실패 로그 수집
   - 원인 분류(환경/테스트/로직/데이터)
   - 로컬 `logging/YYYY-MM-DD.md`와 `errorcase/YYYY-MM-DD_*.md` 기록 후, 공유가 필요한 내용은 정제 문서에 반영
4. 훅 검증: `.githooks/pre-push`
5. 계획 대비 누락/변경점 점검 후 실행 여부 결정

## 4) 누락/변경점 점검 체크리스트

- [ ] 품질 게이트 결과 확보
- [ ] E2E 단독 결과 확보
- [ ] 장애 시 원인/해결/재검증 기록 체계 확인
- [ ] 훅 기반 사전 차단 동작 확인
- [ ] 문서화(작업서/로그) 완료

## 5) 의사결정 기준

- 모든 테스트 통과 + 훅 정상 동작 시: 코드 수정 없이 운영 유지, 기록만 업데이트
- 테스트 실패 시: 원인 해결 후 재검증까지 완료해야 종료
