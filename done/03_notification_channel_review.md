# 알림 채널 통합 검토 및 아키텍처 수립

현재 본 프로젝트는 `email_sender.py`를 통해 이메일 발송 기능을 지원하고 있으나, 사용자들의 즉각적인 접근성과 편의성을 높이기 위해 모바일 메신저 연동 확장을 검토하였습니다. (Task 1.1 및 1.2)

## 1. 후보 채널 분석

### A. 카카오톡 알림톡 (Kakao AlimTalk)

- **장점**: 대다수 국내 사용자가 이용하므로 접근성이 매우 높으며 별도 앱 설치나 가입 유도가 불필요.
- **단점**:
  1. 사업자 등록증 필수 (비즈니스 채널 개설)
  2. 사전 템플릿 검수 과정(2~3일 소요) 및 반려 시 수정 필요.
  3. 알리고(Aligo), 팝빌(Popbill) 등 B2B 중계사를 끼고 유료로 발송해야 함.
- **결론**: 아직 개인 프로젝트/MVP 단계이므로, 사업자 등록과 발송 비용이라는 높은 허들을 감안할 때 즉각적인 도입은 부적절함. 대신 추후 B2C 서비스화 시점에 인터페이스에 맞춰 플러그인 형태로 추가 구현하는 방향으로 미룹니다.

### B. 텔레그램 봇 (Telegram Bot)

- **장점**:
  1. BotFather를 통해 즉각적이고 무료로 개인용/채널용 봇 발급 가능.
  2. 문자열 길이의 제약이 거의 없고 마크다운(Markdown) 혹은 HTML 포매팅 적용이 원활함 (주식 시황 등 긴 요약 리포트를 보내기에 최적).
  3. API 연동이 직관적이며 속도가 빠름.
- **단점**: 알림 채널 구독을 위해 사용자가 Telegram 앱을 설치하고 봇을 구독해야 함.
- **결론**: 현재 단계에서 가장 유의미한 채널 개선안으로 판단됩니다. 이를 먼저 핵심 모듈로 도입합니다.

## 2. 클린 아키텍처 기반 알림 시스템 확장 방향

위 검토 결과에 따라, 알림 채널은 계속 늘어날 수 있으므로 (Email -> Telegram -> 추후 Kakao -> Slack Webhook 등), 특정 모듈(예: `email_sender.py`)에 강결합되지 않도록 **전략 패턴(Strategy Pattern)**과 **포트와 어댑터(Ports and Adapters)** 패턴 철학을 반영하여 인터페이스를 설계해야 합니다.

- **인터페이스 (Protocol 적용)**: `NotificationSender` 역할을 하는 Base Class를 두고, 이메일, 텔레그램 클래스는 이를 개별 구현(Implement)하도록 나눕니다.
- **사용자 의존성**: Notion Database의 컬럼에 수신 희망 채널(예: `Email`, `Telegram`)을 다중 선택하게 하여, `main.py`가 타겟 사용자 정보(DTO)를 받았을 때 각 채널 어댑터를 루핑하며 발송하도록 확장합니다.
