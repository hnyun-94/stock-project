# 역할 체계 확장 계획서

## 요청 요약

- 역할을 부여하는 `.md`, `skills`, `hook`에 `품질 담당자`, `시니어 개발자`, `주식 전문가`를 추가합니다.
- 문서 설명과 실제 검증 기준이 어긋나지 않도록 같은 역할 집합을 공통 규칙으로 통일합니다.

## 역할별 검토

### PM

- 판단: 역할 확장은 반복 작업 품질을 높이는 구조 개선으로 타당합니다.
- 근거: 사용자는 여러 번 다각도 검토를 요구했고, 현재 체계는 제품/기술/운영 중심이라 품질과 도메인 판단이 상대적으로 약합니다.

### TPM

- 판단: 문서만 수정하면 실제 강제가 되지 않으므로 hook 검증까지 함께 바꿔야 합니다.
- 근거: 현재 `pre-push`는 `scripts/check_review_policy.sh`를 호출하고 있어, 역할 집합이 여기와 다르면 규칙이 무력화됩니다.

### 기획자

- 판단: 역할 순서를 고정하고 각 역할 책임을 명시해야 사용자와 에이전트 모두 일관되게 사용합니다.
- 근거: 같은 역할이라도 문서마다 순서와 의미가 다르면 계획서와 PR 본문이 흔들려 가독성과 재사용성이 떨어집니다.

### 시니어 개발자

- 판단: 시니어 개발자 역할은 아키텍처 적합성, 대안 비교, 기술 부채 통제를 담당하도록 정의하는 것이 적절합니다.
- 근거: 기존 `개발자` 역할에 구현 세부와 설계 검토가 함께 섞여 있어, 깊은 기술 판단을 분리하는 편이 유지보수에 유리합니다.

### 개발자

- 판단: 구현 관점은 기존대로 변경 범위, 테스트 가능성, 모듈 경계를 중심으로 유지합니다.
- 근거: 새 역할을 추가하더라도 구현자 관점은 여전히 실질적인 코드 변경 영향과 테스트 작성성을 담당해야 합니다.

### 품질 담당자

- 판단: 품질 담당자는 회귀 위험, 검증 누락, 테스트 전략을 별도 점검하는 역할로 추가해야 합니다.
- 근거: 현재 체계는 테스트 실행은 강하지만, 어떤 회귀가 가능한지 구조적으로 점검하는 역할 명시가 부족합니다.

### 신입개발자

- 판단: 신입개발자 역할은 계속 유지해야 합니다.
- 근거: 새 역할을 넣더라도 문서와 변경 이유를 쉽게 이해할 수 있어야 온보딩 비용이 낮아집니다.

### 운영자

- 판단: 운영자 역할은 관측성, 장애 복구, 운영 절차 검토를 계속 담당해야 합니다.
- 근거: hook와 검증 스크립트 변경은 운영 실패 시 원인 파악 가능성과 직결됩니다.

### 주식 전문가

- 판단: 주식 전문가 역할은 시장 데이터 해석, 투자 표현의 타당성, 도메인 왜곡 여부를 검토하도록 추가해야 합니다.
- 근거: 이 프로젝트는 주식 리포트를 만드는 시스템이므로, 기술적으로 맞아도 시장 해석이 어색하면 사용자 가치가 떨어집니다.

## 검토 라운드

### 1차 검토

- 검토 내용: 어떤 파일이 실제 역할 집합을 정의하는지 추적했습니다.
- 선택/기각 이유: `AGENTS.md`, 관련 skill, `check_review_policy.sh`, 템플릿을 우선 수정 대상으로 선택했고, 전체 과거 문서 일괄 수정은 현재 검증 대상이 아니므로 기각했습니다.

### 2차 검토

- 검토 내용: 새 역할을 어디까지 강제할지 검토했습니다.
- 선택/기각 이유: `pre-push`와 연결된 리뷰 정책 검증을 새 역할 기준으로 강화하기로 했고, 커밋 메시지 hook까지 확장하는 안은 역할 강제와 직접 관련이 약해 제외했습니다.

### 3차 검토

- 검토 내용: 역할 순서와 책임 정의를 정리했습니다.
- 선택/기각 이유: `PM -> TPM -> 기획자 -> 시니어 개발자 -> 개발자 -> 품질 담당자 -> 신입개발자 -> 운영자 -> 주식 전문가` 순서를 채택했고, 문서마다 다른 순서를 쓰는 안은 재사용성과 자동 검증에 불리해 기각했습니다.

## 종합 판단

- 최종 선택안: 문서, skill, hook 검증 기준을 모두 9개 역할 체계로 통일합니다.
- 이유: 역할 정의와 검증 로직이 한 번에 바뀌어야 이후 계획서와 PR에서 누락 없이 재사용할 수 있습니다.
- 남은 리스크: 기존에 작성된 과거 계획서와 PR 예시는 6개 역할 기준일 수 있으나, 변경 파일만 검사하므로 즉시 운영 리스크는 낮습니다.

## 병렬 스트림

- 스트림 A: `AGENTS.md`, 템플릿, 보조 문서에서 역할 정의와 설명 갱신
- 스트림 B: `skills`와 `hook` 검증 기준 갱신

## 수용 기준

- 기준 1: `AGENTS.md`와 skill 문서에서 9개 역할 체계를 일관되게 설명한다.
- 기준 2: `scripts/check_review_policy.sh`가 9개 역할 누락을 검출한다.
- 기준 3: 역할 템플릿 문서가 9개 역할 섹션을 제공한다.
