# 구독자 친화 리포트 레이아웃 개선 계획서

## 요청 요약

- 첨부된 레퍼런스 메일 구조를 참고해 현재 리포트의 구성과 배치를 더 읽기 쉽게 개선합니다.
- 목표는 정보량을 줄이는 것이 아니라, 상단에서 결론을 빠르게 읽고 중단에서 판단 근거를 구조적으로 이해하며 하단에서 관심 테마와 보유 종목을 선별적으로 읽을 수 있게 만드는 것입니다.
- Markdown 기반 출력과 이메일 HTML 변환 모두에서 가독성이 좋아져야 합니다.

## 역할별 검토

### PM

- 판단: 상단 요약, 중단 맥락, 하단 상세의 3단 구조로 바꾸는 것이 사용자 가치에 직접적입니다.
- 근거: 현재 리포트는 섹션마다 같은 카드 패턴이 반복돼 “무엇이 가장 중요한지”가 잘 안 보입니다.

### TPM

- 판단: builder와 formatter 역할을 더 선명히 분리해야 합니다.
- 근거: 지금은 builder가 카드 데이터를 만들고 formatter가 거의 동일한 템플릿으로 반복 렌더링해 레이아웃 차별성이 약합니다.

### 기획자

- 판단: 레퍼런스 메일의 핵심은 “상단 요약 바 + 중단 3개 렌즈 + 하단 상세 카드”입니다.
- 근거: 사용자는 메일을 처음 열었을 때 10초 안에 큰 그림을 파악하고, 이후 필요한 블록만 읽는 경향이 있습니다.

### 시니어 개발자

- 판단: top summary, scoreboard, insight lenses 같은 중간 구조는 payload에서 명시적으로 제공하는 편이 유지보수에 유리합니다.
- 근거: formatter만으로 억지로 조합하면 렌더링 계층에 판단 로직이 새로 섞입니다.

### 개발자

- 판단: `report_builder.py`에 레이아웃용 파생 데이터 구조를 추가하고, `report_formatter.py`에는 섹션별 전용 렌더러를 두는 방식이 적절합니다.
- 근거: 기존 generic card 템플릿은 유지하되, 빠른 판단용 섹션은 별도 renderer로 분기하는 편이 구현 리스크가 낮습니다.

### 품질 담당자

- 판단: 리포트 구조가 바뀌면 테스트도 “키 존재”보다 “새 섹션의 존재와 압축 구조”를 검증해야 합니다.
- 근거: 현재 테스트는 기존 섹션 헤더 위주라, 실제 구독자 친화성이 악화돼도 감지하지 못합니다.

### 신입개발자

- 판단: 구조 이름 자체가 역할을 드러내야 합니다.
- 근거: `지금 결론`, `오늘 바로 볼 숫자`, `세 가지 축`, `시간대 압축판` 같은 이름이 읽는 순서를 자연스럽게 안내합니다.

### 운영자

- 판단: Markdown과 HTML 둘 다에서 안정적으로 보여야 하므로 표와 blockquote 스타일을 같이 보강해야 합니다.
- 근거: 이메일 렌더러는 CSS 지원이 제한적이어서 구조가 단순하고 반복 가능한 표/박스 위주가 더 안전합니다.

### 주식 전문가

- 판단: “매크로 / 실적·수급 / 이벤트”의 분리는 투자 판단 흐름과 잘 맞습니다.
- 근거: 실제 판단은 거시 변수, 실적·자금 흐름, 단기 이벤트를 따로 보되 마지막에 종합하는 방식으로 이뤄집니다.

### 프롬프트 전문가

- 판단: 상위 섹션은 정보 우선순위가 드러나는 이름과 짧은 요약을 우선 제공해야 합니다.
- 근거: 리포트는 완전한 보고서보다 의사결정 프롬프트에 가까워, 빠른 스캐닝이 가능해야 합니다.

### 컨텍스트 엔지니어

- 판단: 반복 서술을 줄이고 재사용 가능한 압축 구조를 만들면 출력 토큰과 읽기 비용을 동시에 줄일 수 있습니다.
- 근거: 같은 `긍정/중립/부정`과 `왜 중요한가`가 매 섹션마다 장황하게 반복되면 컨텍스트 효율이 나빠집니다.

## 검토 라운드

### 1차 검토

- 검토 내용: 첨부 레퍼런스 메일과 현재 리포트를 비교해 구조 차이를 정리했습니다.
- 선택/기각 이유: `상단 결론 / 중단 맥락 / 하단 상세` 구조를 채택했고, 현재처럼 모든 섹션을 동일 카드로 반복하는 구조는 기각했습니다.

### 2차 검토

- 검토 내용: builder와 formatter 중 어디에 무엇을 둘지 정했습니다.
- 선택/기각 이유: 파생 데이터 구조(`decision_tiles`, `market_scoreboard`, `insight_lenses`)는 builder에 두고, 실제 배치는 formatter에 두는 쪽을 선택했습니다.

### 3차 검토

- 검토 내용: Markdown 기반에서 얼마나 시각적 계층을 살릴 수 있는지 검토했습니다.
- 선택/기각 이유: 이메일 호환성과 구현 안정성을 고려해 Markdown table + blockquote + CSS 보강을 중심으로 가고, 과도한 raw HTML 레이아웃 의존은 기각했습니다.

## 종합 판단

- 최종 선택안: 리포트를 `지금 결론 -> 오늘 바로 볼 숫자 -> 세 가지 축 -> 시간대 압축판 -> 테마/보유 종목 상세 -> 장기 플랜 -> 용어 풀이` 구조로 재편합니다.
- 이유: 첨부 레퍼런스의 정보 계층을 가장 현실적으로 재현하면서도 기존 payload/테스트 구조와 충돌이 적기 때문입니다.
- 남은 리스크: Markdown 기반이라 완전한 카드형 시각 디자인은 제한되지만, 섹션 구분과 반복 문장 압축만으로도 체감 가독성은 크게 좋아질 가능성이 높습니다.

## 병렬 스트림

- 스트림 A: builder 파생 데이터 구조 추가 (`decision_tiles`, `market_scoreboard`, `insight_lenses`)
- 스트림 B: formatter 섹션 재배치 + CSS 개선 + 테스트 갱신

## 커밋 예산

- 커밋 1: 계획서 + builder 구조 개선, 목표 280~340줄
- 커밋 2: formatter/CSS/테스트 개선, 목표 220~320줄

## 수용 기준

- 기준 1: 리포트 상단에서 결론, 우선 테마, 오늘 액션이 빠르게 보인다.
- 기준 2: 중단에는 매크로/실적·수급/이벤트의 3축 정리가 존재한다.
- 기준 3: 시간대별 판단은 압축된 표나 요약 구조로 제공된다.
- 기준 4: 보유 종목과 테마 섹션은 동일 generic card 반복보다 더 압축되고 구독자 친화적으로 렌더링된다.

## 실행 결과

- 상단 구조를 `지금 결론 -> 빠르게 보는 판단표 -> 오늘 바로 볼 숫자` 순서로 재배치했습니다.
- `report_builder.py`에 `decision_tiles`, `market_scoreboard`, `insight_lenses`를 추가해 formatter가 목적별 레이아웃을 안정적으로 렌더링할 수 있게 했습니다.
- `report_formatter.py`는 기존 generic card 중심 렌더링에서 벗어나 상단 대시보드, 3축 브리핑, 시간대 압축판, 테마/보유 종목 compact brief 구조로 재구성했습니다.
- 이메일 HTML 스타일은 표, blockquote, section background를 보강해 첨부 레퍼런스 메일처럼 위에서 아래로 스캔하기 쉬운 형태로 조정했습니다.
- 회귀 테스트는 `tests/services/test_report_builder.py`, `tests/test_e2e_dryrun.py`를 갱신했고, `uv run python -m pytest tests/services/test_report_builder.py tests/test_e2e_dryrun.py -q` 및 `sh scripts/run_quality_gate.sh --range origin/master..HEAD`를 통과했습니다.
