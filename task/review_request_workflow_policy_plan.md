# 제안·검토 요청 기본 워크플로우 정책 반영 계획서

## 요청 요약

- 사용자가 제안하거나 검토를 요청할 때는 기존 역할들을 활용해 다양하고 반복적인 검토를 먼저 진행한 뒤 계획서를 작성하는 흐름을 기본값으로 고정합니다.

## 역할별 검토

### PM

- 판단: 채택
- 근거 또는 이유: 검토형 요청에서 바로 구현으로 들어가면 사용자가 기대한 비교·선택 과정이 생략될 수 있습니다.

### TPM

- 판단: `AGENTS.md`와 skill trigger를 함께 수정
- 근거 또는 이유: 상위 계약서와 실제 실행 스킬이 같이 바뀌어야 drift가 줄어듭니다.

### 기획자

- 판단: 제안/검토 요청은 구현보다 먼저 계획서 산출이 나오도록 기본 흐름을 명시
- 근거 또는 이유: 사용자 기대치를 안정적으로 맞출 수 있습니다.

### 시니어 개발자

- 판단: 반복 요구 승격 대상으로 적절
- 근거 또는 이유: ad-hoc 답변보다 운영 규칙으로 승격하는 편이 일관성이 높습니다.

### 개발자

- 판단: 문서 2곳 수정으로 충분
- 근거 또는 이유: `AGENTS.md`, `review-driven-execution` skill이 실제 라우팅 기준입니다.

### 품질 담당자

- 판단: guardrail 문구를 함께 보강
- 근거 또는 이유: trigger만 넓히면 누락 가능성이 남습니다.

### 신입개발자

- 판단: 제안/검토 요청도 기본 흐름이 명시되면 언제 계획서를 먼저 써야 하는지 이해하기 쉽습니다.
- 근거 또는 이유: 암묵 규칙보다 명시 규칙이 온보딩 난도를 낮춥니다.

### 운영자

- 판단: 사람마다 해석이 달라지는 일을 줄이는 운영 규칙으로 적절합니다.
- 근거 또는 이유: 요청 유형별 기본 흐름이 고정되면 세션 간 편차가 줄어듭니다.

### 주식 전문가

- 판단: 제안/검토 요청은 도메인 해석 비교가 중요하므로 반복 검토가 유리합니다.
- 근거 또는 이유: 단일 시각으로 빠르게 결론내리면 해석 폭이 좁아질 수 있습니다.

### UI/UX 전문가

- 판단: 계획서 선행은 사용자에게 검토 과정이 보이는 구조를 만들어 줍니다.
- 근거 또는 이유: 결과만 주는 것보다 판단 경로가 드러나는 쪽이 기대와 맞습니다.

### 퍼블리셔 전문가

- 판단: 계약서와 skill 양쪽에 같은 메시지가 있어야 실제 사용 시 drift가 줄어듭니다.
- 근거 또는 이유: 한쪽만 수정하면 문서 기준과 실행 기준이 어긋날 수 있습니다.

## 구현 범위

- `AGENTS.md` Review Model에 제안/검토 요청 기본 처리 규칙 추가
- `.agents/skills/review-driven-execution/SKILL.md` trigger/guardrail 보강

## 수용 기준

- 사용자가 제안/검토 요청을 할 때 `review-driven-execution`이 기본 흐름으로 해석된다.
- 반복 역할 검토 후 계획서를 작성한다는 문구가 계약서와 skill 양쪽에 남는다.

## 종합 판단

- 이번 요청은 반복적으로 쓰일 작업 방식 규칙이므로 ad-hoc 답변이 아니라 계약서/skill 기준선으로 승격하는 것이 맞습니다.
- 따라서 제안·검토 요청은 앞으로 기본적으로 다중 역할 반복 검토 후 계획서 작성으로 이어지도록 고정합니다.
