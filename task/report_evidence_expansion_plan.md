# 리포트 근거 확장 및 중복 축소 계획서

## 요청 요약

- 현재 리포트는 섹션 간 문장 패턴이 반복되고, 근거로 제시되는 데이터가 적으며, 일부 섹션은 주제와 직접 맞닿은 정보가 부족합니다.
- 테마/보유 종목/시장 섹션마다 근거를 3~5개 수준으로 더 풍부하게 보여주고, 수집 데이터 범위와 품질을 함께 개선합니다.

## 역할별 검토

### PM

- 판단: 채택
- 근거 또는 이유: 사용자는 "무슨 말을 하는지"보다 "왜 그렇게 말하는지"를 더 알고 싶어 합니다. 근거 확장이 만족도와 신뢰도에 직접 연결됩니다.

### TPM

- 판단: 수집 확대와 출력 개선을 동시에 진행하되, 런타임 증가는 완만하게 제한
- 근거 또는 이유: 기사 수집/리드 문단 추출을 무작정 크게 늘리면 호출량과 지연이 급증하므로, 검색 폭은 소폭 확대하고 주제 적합도 필터를 함께 넣어야 합니다.

### 기획자

- 판단: 기존 섹션은 유지하고 근거 밀도와 주제 적합도를 높이는 방향이 적절
- 근거 또는 이유: 사용자는 이미 현재 리포트 구조에 익숙하므로, 새 섹션을 늘리기보다 각 섹션이 "다른 이야기를 하고 있다"는 느낌을 주는 편이 낫습니다.

### 시니어 개발자

- 판단: `topic_news`에서 주제 적합도를 높이고, `report_builder`에서 공통 템플릿 문장 반복을 줄여야 함
- 근거 또는 이유: 입력이 부정확하면 AI와 builder 모두 흔들리고, 출력이 공통 템플릿이면 입력이 좋아도 결국 비슷한 문장이 반복됩니다.

### 개발자

- 판단: 1차 구현 범위는 네 갈래
- 근거 또는 이유:
  - 토픽 뉴스 검색 폭 확대
  - 토픽별 관련 커뮤니티 post 선별
  - theme/holding 카드 근거 수 확장
  - subject-specific why/watch/outlook 규칙 추가

### 품질 담당자

- 판단: topic relevance와 formatter 출력 둘 다 테스트 필요
- 근거 또는 이유: 이번 작업은 데이터 선택과 표현 계층이 동시에 바뀌므로 한쪽만 검사하면 회귀를 놓치기 쉽습니다.

### 신입개발자

- 판단: 주제별 alias/근거 규칙을 helper로 분리
- 근거 또는 이유: `S&P500`, `인공지능(AI)`, `이차전지`처럼 특성화된 주제가 늘어날수록 if 문 산재보다 맵과 helper가 유지보수에 낫습니다.

### 운영자

- 판단: 외부 호출 수는 단계적으로만 확대
- 근거 또는 이유: 검색 결과 개수와 리드 문단 추출 개수를 조금 늘리는 정도는 감당 가능하지만, 소스 수 추가나 무제한 확장은 장애 포인트가 됩니다.

### 주식 전문가

- 판단: 주제별 근거는 서로 다른 축으로 보여야 함
- 근거 또는 이유: 이차전지 섹션에 AI 일반론이 반복되면 해석이 흐려집니다. 테마별로 수요, 정책, 가격, 수급, 실적 같은 고유 축이 드러나야 합니다.

### UI/UX 전문가

- 판단: compact brief에서 근거/링크를 더 많이 보여주되 가독성을 해치지 않게 줄 단위로 분리
- 근거 또는 이유: 한 줄에 근거 2개를 `/`로 이어 붙이는 방식은 압축은 되지만 읽기 피로가 큽니다.

### 퍼블리셔 전문가

- 판단: compact brief를 한 줄 요약 중심에서 다중 flat bullet 구조로 약간 풀어주는 편이 낫다
- 근거 또는 이유: 이메일 환경에서는 긴 inline 문장보다 짧은 여러 줄이 스캔성이 더 좋습니다.

## 검토 라운드

### 1차 검토

- 선택안 A: builder 문장만 손봐서 반복 줄이기
- 선택안 B: 데이터 선택 + builder 문장 + formatter 노출량을 함께 조정
- 판단: B 채택
- 기각 이유:
  - A는 입력 근거가 빈약한 문제를 그대로 둬서 개선 폭이 작습니다.

### 2차 검토

- 검토 내용: 수집 범위 확대 방식
- 판단: 소스 수 추가보다 기존 3소스의 depth 확대 + 주제 적합도 필터 + 관련 커뮤니티 선별을 우선
- 기각 이유:
  - 새 소스 추가는 호출 정책/테스트/운영 리스크가 커 이번 라운드 범위를 넘깁니다.

### 3차 검토

- 검토 내용: 반복 축소 방식
- 판단: subject-specific why/watch/outlook 사전을 추가하고 evidence-first 문장으로 전환
- 기각 이유:
  - 단순 랜덤 문구 치환은 반복만 숨길 뿐, 주제 특성화와 근거 연결이 약합니다.

## 선택안과 기각 이유

### 채택

- 토픽 뉴스 검색 수를 소폭 확대
- 토픽별 관련 뉴스/커뮤니티 post 선별 helper 추가
- theme/holding 카드 근거와 관련 링크를 3~5개 수준으로 확장
- `왜 중요한가`, `지금 볼 것`, `다음 체크포인트`, `세 가지 시각`에 subject-specific 규칙 추가
- compact formatter에서 근거/링크를 여러 줄로 노출

### 이번 라운드 기각

- 신규 뉴스 소스 추가
- DB 스키마 확장
- 대형 새 섹션 추가
- AI 프롬프트 전면 개편

## 병렬 스트림

1. Stream A - 수집 근거 확장
- `src/services/topic_news.py`
- `src/main.py`

2. Stream B - builder 중복 축소
- `src/services/report_builder.py`

3. Stream C - formatter / 테스트
- `src/utils/report_formatter.py`
- `tests/services/`
- `tests/test_e2e_dryrun.py`

## 커밋 예산

1. topic relevance + 커뮤니티 입력 확장 + 계획서: 260 lines 이하
2. builder/formatter 중복 축소 + 회귀 테스트: 360 lines 이하

## 구현 범위

- `src/services/topic_news.py`
  - 주제 alias/적합도 helper 추가
  - 뉴스/커뮤니티 선별 보강
  - 기사 수집 depth 소폭 확대
- `src/main.py`
  - theme batch에 주제별 커뮤니티 post 전달
- `src/services/report_builder.py`
  - theme/holding 근거 3~5개 확장
  - subject-specific why/watch/outlook 보강
  - evidence-first context 문장으로 반복 축소
- `src/utils/report_formatter.py`
  - compact brief를 다중 근거/링크 노출 구조로 조정
- 테스트
  - topic relevance
  - theme/holding 근거 수와 subject-specific 표현
  - markdown/e2e 출력

## 후속 범위

- 추가 뉴스 소스 확대 여부 재검토
- 보유 종목별 외부 지표(OpenDART/FRED/SEC) 직접 연결
- theme/holding별 근거 카드 전용 UI 분리

## 수용 기준

- theme/holding 섹션이 이전보다 더 적은 반복 문장으로 구성된다.
- 각 섹션에서 근거가 3~5개 수준으로 노출된다.
- 토픽/보유 종목과 무관한 저품질 뉴스 비중이 줄어든다.
- 관련 커뮤니티 반응이 theme 분석 입력에 반영된다.
- 표준 품질 게이트를 통과한다.

## 종합 판단

- 이번 문제는 문장만의 문제가 아니라 입력 데이터와 표현 계층이 함께 얽혀 있습니다.
- 따라서 topic relevance, evidence density, subject specificity를 동시에 손보는 것이 가장 효과적입니다.
- 이번 라운드는 기존 3소스 체계를 유지하면서도 사용자 체감 품질을 크게 올릴 수 있는 최적 범위입니다.
