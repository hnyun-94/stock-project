- 요약 문구에 `...` 또는 `…`로 의미가 잘린 문장이 남지 않도록 문장 완결성을 보장합니다.

## 역할별 검토

### PM

- 판단: 이번 요청은 기능 추가보다 사용자 체감 품질 개선에 직접 연결되는 수정입니다.
- 근거 또는 이유: 동일한 데이터라도 읽는 피로도와 첫인상이 달라지면 만족도가 크게 달라집니다.

### TPM

- 판단: 색상, 헤더, 문장 정책은 서로 다른 레이어에서 다뤄야 합니다.
- 근거 또는 이유: 색상/표현은 formatter, 문장 절단은 builder에 있어 책임 분리가 명확해야 회귀를 막기 쉽습니다.

### 기획자

- 판단: 표 헤더는 시스템 용어보다 독자 관점 언어가 낫습니다.
- 근거 또는 이유: `현재 값`, `1D 변화` 같은 표현보다 `오늘 숫자`, `하루 변화`가 빠르게 읽힙니다.

### 시니어 개발자

- 판단: 공통 truncation 규칙을 문장 단위 종료 방식으로 바꾸는 것이 맞습니다.
- 근거 또는 이유: 개별 카드마다 예외 처리하기보다 공통 helper를 고치면 전체 출력 정책을 일관되게 가져갈 수 있습니다.

### 개발자

- 판단: 테스트는 formatter alias와 report builder truncation 둘 다 고정해야 합니다.
- 근거 또는 이유: 이번 문제는 UI와 내용 생성이 동시에 연결된 회귀 포인트입니다.

### 품질 담당자

- 판단: `…` 미노출과 색상 토큰 반영 여부를 테스트로 남겨야 합니다.
- 근거 또는 이유: 사람이 직접 보기 전에도 깨진 요약과 잘못된 팔레트를 빠르게 탐지할 수 있어야 합니다.

### 신입개발자

- 판단: 헤더 alias는 매핑 테이블로 두는 편이 읽기 쉽습니다.
- 근거 또는 이유: 나중에 표현을 바꿀 때 formatter 한 곳만 보면 됩니다.

### 운영자

- 판단: 이번 변경은 외부 연동보다 출력 안정성 중심이므로 운영 리스크가 낮습니다.
- 근거 또는 이유: 메일 렌더링과 텍스트 가공 규칙 변경은 배포 리스크가 비교적 제한적입니다.

### 주식 전문가

- 판단: 요약 문장은 끝맺음이 있어야 투자 판단 근거를 오해하지 않습니다.
- 근거 또는 이유: 핵심 문장이 중간에서 잘리면 의미가 반대로 읽히거나 과도하게 불안하게 보일 수 있습니다.

### UI/UX 전문가

- 판단: 메인/보조색 요청을 헤드라인, 섹션 타이틀, 강조 박스, 링크에 일관되게 적용하는 것이 적절합니다.
- 근거 또는 이유: 포인트 색이 흩어지면 브랜드 톤이 아니라 산만한 화면처럼 보입니다.

### 퍼블리셔 전문가

- 판단: 이메일은 CSS 변수보다 고정된 인라인 컬러가 안전합니다.
- 근거 또는 이유: 메일 클라이언트 호환성을 생각하면 직접 색상을 주입하는 방식이 더 보수적입니다.

## 종합 판단

- formatter에는 팔레트 토큰과 헤더 alias를 추가하고, builder에는 완결된 문장 우선 truncation을 적용합니다.
- 테스트는 `색상 반영`, `친화적 헤더`, `ellipsis 제거`를 기준으로 보강합니다.
