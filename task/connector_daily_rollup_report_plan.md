# 외부 커넥터 일자별 집계 리포트 계획서

## 요청 배경

- 미반영 장애는 현재 확인되지 않았다.
- 남아 있는 최우선 개발 항목은 `external_connector_runs` 일자별 집계 리포트 배치 생성이다.
- 이미 source별 실시간 알림은 구현됐지만, 최근 7일 품질 추세를 한 번에 읽을 수 있는 리포트 계층은 없다.

## 목표

- `external_connector_runs`를 일자별로 집계해 source별 성공률, 평균 지연, 실패 건수를 최근 7일 기준으로 조회할 수 있게 한다.
- 이 집계를 사용자/운영자가 읽을 수 있는 카드형 리포트로 변환한다.
- 리포트에는 `짧은 판단 + 근거 + 긍정/중립/부정 시각 + 최근 일자별 표`가 함께 포함되도록 한다.

## 역할별 검토

### PM

- 판단: 채택
- 근거: 사용자 입장에서는 "이 리포트를 얼마나 믿어도 되는가"가 중요하다. 데이터 품질을 숫자로 보여주면 판단 비용이 줄어든다.

### TPM

- 판단: 채택
- 근거: 현재는 성공률 단일 수치만 써서 시간축 정보가 없다. 일자별 롤업이 있어야 장애가 일시적인지 구조적인지 구분할 수 있다.

### 기획자

- 판단: 채택
- 근거: 운영용 숫자를 그대로 노출하면 읽기 어렵다. 한 줄 판단과 근거, 표를 묶은 카드 구조가 사용성과 설명력을 동시에 만족한다.

### 개발자

- 판단: 채택
- 근거: 집계는 DB가 담당하고, 가치 판단 문장은 report_builder가 담당하는 현재 아키텍처와 잘 맞는다.

### 신입개발자

- 판단: 채택
- 근거: `DB 집계 -> payload 카드 -> formatter 렌더링` 구조는 테스트 지점이 분명하다.

### 운영자

- 판단: 채택
- 근거: 최근 오류 사유와 일자별 성공률이 같이 보이면 "오늘만 문제인지", "계속 나빠지는지"를 빠르게 판단할 수 있다.

## 검토 라운드

### 1차 검토

- 선택안 A: 운영용 별도 스크립트만 추가하고 사용자 리포트에는 노출하지 않음
- 선택안 B: DB 롤업을 만들고, 리포트 카드로도 함께 노출
- 판단: B 채택
- 기각 이유:
  - A는 배치 산출물은 생기지만 실제 사용자/운영 가치가 즉시 드러나지 않는다.

### 2차 검토

- 검토 내용: 집계 저장 방식
- 판단: 우선은 materialized table 없이 `external_connector_runs`에서 최근 7일 롤업 조회
- 기각 이유:
  - 현 단계에서는 데이터량이 작고, 새 테이블을 추가하는 비용보다 조회 함수 추가가 단순하다.

### 3차 검토

- 검토 내용: 리포트 표현 방식
- 판단: `데이터 신뢰도` 섹션을 추가하고, 카드 아래에 최근 일자별 표를 함께 렌더링
- 기각 이유:
  - 기존 1W/1M 문장에 숫자를 억지로 섞으면 읽기성만 떨어지고 일자별 집계라는 목표도 흐려진다.

## 구현 범위

1. `src/utils/database.py`
- 최근 N일 source별 일자 롤업 조회 메서드 추가
- 최근 실패 사유 조회 메서드 추가

2. `src/services/report_builder.py`
- 커넥터 품질 카드 생성 로직 추가
- payload에 `data_quality_section` 포함

3. `src/utils/report_formatter.py`
- 데이터 신뢰도 섹션과 일자별 표 렌더링 추가

4. `src/main.py`
- DB에서 최근 7일 롤업과 최근 실패 사유를 조회해 builder로 전달

5. 테스트 및 문서
- DB 집계 테스트
- report builder / formatter 테스트
- README, todo, roadmap 동기화

## 커밋 예산

1. 계획서 + DB 롤업 + DB 테스트: 220 lines 이하
2. 리포트 카드/렌더링 + 관련 테스트: 330 lines 이하
3. main 연결 + 문서 동기화: 160 lines 이하

## 완료 기준

- 최근 7일 일자별 source 성공률/평균 지연/실패 건수가 재현된다.
- 최종 리포트에 `데이터 신뢰도` 섹션이 생기고, 일자별 집계 표가 포함된다.
- 실패 사유 요약이 카드의 근거로 노출된다.
- `uv run python -m pytest tests/services/ tests/test_e2e_dryrun.py -q` 통과
- `pre-push` 및 품질 게이트 통과

## 종합 판단

- 실시간 운영 알림 다음 단계로 가장 가치가 큰 작업은 일자별 품질 리포트화다.
- 새 저장소를 만들기보다 기존 텔레메트리와 리포트 구조를 재사용하는 편이 구현 비용 대비 효과가 크다.
- 따라서 이번 작업은 DB 롤업과 사용자 가시화까지 한 번에 구현한다.
