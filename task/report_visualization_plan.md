# 리포트 지수/지표 시각화 계획서

## 요청 요약

- 지수와 지표를 리포트에서 더 빨리 읽을 수 있도록 시각화 방법을 구체적으로 검토하고 구현합니다.
- 현재 운영 경로가 이메일 HTML이므로, 메일 클라이언트 호환성을 해치지 않는 방식만 채택합니다.

## 역할별 검토

### PM

- 판단: 채택
- 근거 또는 이유: 숫자가 많아질수록 사용자 체감 품질은 "더 많이 보여주는가"보다 "더 빨리 읽히는가"에 좌우됩니다.

### TPM

- 판단: 텍스트 기반 미니 시각화 우선
- 근거 또는 이유: 현재 렌더링 경로는 Markdown -> 인라인 HTML이며, JS 차트나 외부 이미지 의존은 운영 리스크가 큽니다.

### 기획자

- 판단: 기존 섹션 구조 유지
- 근거 또는 이유: `오늘 바로 볼 숫자`, `데이터 신뢰도`, `보조 지표 해석`은 이미 정보 밀도가 높아 새로운 대형 차트 섹션보다 기존 표/카드 내부 강화가 자연스럽습니다.

### 시니어 개발자

- 판단: builder에서 시각화 문자열을 만들고 formatter는 렌더링만 담당
- 근거 또는 이유: 시각화 규칙이 데이터 의미와 연결돼 있으므로 payload 조립 단계에서 일관되게 계산하는 편이 테스트와 재사용에 유리합니다.

### 개발자

- 판단: 1차 구현 범위는 신뢰도 게이지, 심리 게이지, 성공률 바, delta 방향표시, 3포인트 스파크라인
- 근거 또는 이유: 현재 데이터만으로 즉시 구현 가능하고 DB 스키마나 크롤러 확장 없이도 효과가 큽니다.

### 품질 담당자

- 판단: 회귀 테스트 필수
- 근거 또는 이유: 메일 HTML은 작은 문자열 변경도 깨지기 쉬워 markdown/HTML 출력과 payload 양쪽을 모두 고정해야 합니다.

### 신입개발자

- 판단: 공통 helper 명시화
- 근거 또는 이유: bar, sparkline, delta marker 규칙이 함수로 분리돼 있으면 후속 수정 경로가 명확합니다.

### 운영자

- 판단: 외부 리소스 없는 시각화만 채택
- 근거 또는 이유: CID 이미지, 외부 이미지 링크, 클라이언트별 스크립트 차트는 장애 원인과 디버깅 포인트를 늘립니다.

### 주식 전문가

- 판단: 색보다 방향과 강도를 우선 표현
- 근거 또는 이유: 투자 해석은 빨강/파랑 색상보다 `상승/하락/보합`, `1D/7D` 방향이 먼저 명확해야 왜곡이 적습니다.

### UI/UX 전문가

- 판단: 큰 차트보다 미니 시각화
- 근거 또는 이유: 현재 메일은 카드와 표가 연속되는 읽기 구조라, 작은 시각화가 스캔성을 높이고 모바일 폭도 덜 침범합니다.

### 퍼블리셔 전문가

- 판단: 유니코드 기반 시각화 우선
- 근거 또는 이유: `▲`, `▼`, `▁▂▃`, `▓░`는 메일 클라이언트 호환성이 높고 별도 CSS 의존이 낮습니다.

## 검토 라운드

### 1차 검토

- 선택안 A: 이미지 차트 또는 외부 차트 링크 추가
- 선택안 B: 표/카드 내부 텍스트 기반 미니 시각화 추가
- 판단: B 채택
- 기각 이유:
  - A는 메일 클라이언트 이미지 차단과 모바일 레이아웃 리스크가 큽니다.

### 2차 검토

- 검토 내용: 시각화 데이터 출처
- 판단: 기존 payload와 현재 전달되는 trend/delta 값만 우선 사용
- 기각 이유:
  - 지수용 장기 시계열 저장까지 이번 라운드에 포함하면 DB/크롤러/런타임 범위가 커집니다.

### 3차 검토

- 검토 내용: 표현 방식
- 판단:
  - 신뢰도/심리: 게이지 바
  - 데이터 품질: 성공률 바 + 지연 상태칩
  - 도메인 지표: 방향 아이콘 + 3포인트 스파크라인
- 기각 이유:
  - 색상만으로 상승/하락을 구분하는 방식은 메일/접근성 환경에서 안정적이지 않습니다.

## 선택안과 기각 이유

### 채택

- `reliability_badge`에 10칸 게이지 추가
- `market_scoreboard`의 심리 행에 심리 게이지 추가
- `data_quality_section`의 성공률/평균 지연 셀에 bar/chip 추가
- `domain_signal_sections`의 최근값/변화 셀에 sparkline/delta marker 추가

### 이번 라운드 기각

- JS 차트
- 외부 이미지 기반 차트
- SVG/CID 임베드 차트
- 지수 히스토리 DB 신규 저장

## 병렬 스트림

1. Stream A - payload helper
- bar/sparkline/delta helper 추가
- reliability/scoreboard/data_quality/domain signal payload 확장

2. Stream B - formatter 반영
- 게이지/칩/스파크라인이 Markdown과 HTML에서 자연스럽게 보이도록 렌더링 조정

3. Stream C - 테스트/문서
- payload/markdown/html 회귀 테스트 보강
- 계획서와 완료 문서 동기화

## 커밋 예산

1. payload helper + formatter + 테스트 + 계획/완료 문서 동기화: 320 lines 이하

- KOSPI/KOSDAQ/환율/WTI 7일 시계열 저장
- 지수별 실제 미니 추세 차트 고도화
- 운영 대시보드와 이메일 시각화 규칙 공통화

## 수용 기준

- 메일 본문에서 신뢰도, 심리, 데이터 품질, 보조 지표 변화가 숫자만보다 빠르게 읽힌다.
- 외부 리소스 없이 렌더링된다.
- markdown/HTML 테스트가 새 시각화 문자열을 검증한다.
- `tests/services/`와 `tests/test_e2e_dryrun.py`가 통과한다.

## 종합 판단

- 현재 리포트 구조에서는 대형 차트보다 "숫자 옆 미니 시각화"가 맞습니다.
- 이번 라운드는 기존 데이터만 재활용하는 방향으로 효과 대비 리스크가 가장 낮습니다.
- 따라서 payload 기반 텍스트 시각화를 먼저 구현하고, 지수 시계열 저장은 후속 라운드로 분리합니다.
