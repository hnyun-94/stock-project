# AGENTS.md 최적화 계획서

## 요청 요약

- `AGENTS.md`에는 꼭 필요한 주요 지침과 작업 플로우만 남깁니다.
- 구체적인 동작 방식, 세부 절차, 예시, 경계 조건은 가능한 한 `skills`와 `hook`로 이동합니다.
- 결과물은 Codex가 빠르게 읽고 라우팅하기 쉬운 구조여야 합니다.

## 역할별 검토

### PM

- 판단: `AGENTS.md`는 상위 계약서 역할만 하고, 세부 실행서는 skills로 분리하는 편이 사용자 가치에 맞습니다.
- 근거: 큰 파일 하나에 모든 규칙이 섞이면 실제 작업 시작 속도가 느려지고, 반복 작업에서도 중요한 규칙을 찾기 어려워집니다.

### TPM

- 판단: hook에서 강제 가능한 규칙과 문서/skill로만 관리해야 할 규칙을 분리해야 합니다.
- 근거: `pre-push`는 deterministic 검증에 적합하지만, 에이전트 라우팅이나 프롬프트 설계 원칙 같은 내용은 skill 쪽이 적합합니다.

### 기획자

- 판단: `AGENTS.md`는 “무엇을 해야 하는지”와 “어디를 봐야 하는지”만 보여주는 인덱스형 문서가 되어야 합니다.
- 근거: 읽는 순서와 책임 경계를 명확히 해야 Codex가 덜 헤매고, 신규 작업도 일관된 방식으로 시작할 수 있습니다.

### 시니어 개발자

- 판단: 세부 정책을 `AGENTS.md`에 중복 보관하는 현재 구조는 유지보수 비용이 큽니다.
- 근거: 같은 정책이 `AGENTS.md`, skill, hook 설명에 반복되면 한 곳만 갱신되어도 drift가 생깁니다.

### 개발자

- 판단: 기존 skill을 재사용하고, 비어 있는 영역만 새 skill로 채우는 쪽이 구현 효율이 높습니다.
- 근거: 이미 `review-driven-execution`, `pr-governance`, `pr-creator`, `command-permission-governance`, `codex-session-setup`이 있어 전면 신규 설계는 불필요합니다.

### 품질 담당자

- 판단: `AGENTS.md` 축약 후에도 품질 게이트와 리뷰 정책이 흐려지지 않도록 상위 규칙은 유지해야 합니다.
- 근거: “짧게 만들기”가 “중요 규칙 삭제”로 이어지면 오히려 회귀 위험이 커집니다.

### 신입개발자

- 판단: `AGENTS.md`는 길더라도 자세한 설명보다, 어떤 skill을 언제 써야 하는지가 더 중요합니다.
- 근거: 신규 기여자는 세부 절차를 한 파일에서 모두 읽기보다, 시작점과 참조 경로가 명확한 편이 이해하기 쉽습니다.

### 운영자

- 판단: hook가 강제하는 규칙은 `AGENTS.md`에 요약만 남기고, 실제 authoritative source는 hook/script로 둬야 합니다.
- 근거: 운영 기준은 실행 결과가 진실 소스이므로, 문서보다 `.githooks/pre-push`와 `scripts/*`가 우선해야 합니다.

### 주식 전문가

- 판단: 도메인 특성상 `AGENTS.md`에는 투자 리포트 프로젝트라는 맥락과 데이터 해석 책임만 짧게 남기면 충분합니다.
- 근거: 주식 해석 기준의 상세 절차는 리포트 생성 코드와 관련 skill/문서에서 관리하는 편이 더 정확합니다.

### 프롬프트 전문가

- 판단: 최상위 지시서는 저엔트로피 구조여야 하며, 긴 예시와 세부 조건은 하위 문서로 분리해야 합니다.
- 근거: 프롬프트의 효율은 정보량보다 우선순위와 라우팅의 명확성에 좌우되며, 긴 상위 지시서는 핵심 제약을 희석시킬 수 있습니다.

### 컨텍스트 엔지니어

- 판단: `AGENTS.md`는 자주 필요한 고신호 정보만 남기고, 변동 가능성이 큰 세부 절차는 skill/hook로 외부화하는 것이 context 효율에 유리합니다.
- 근거: 상위 문서가 비대해질수록 매 세션마다 같은 긴 문맥을 다시 읽게 되어 토큰 비용과 오염 가능성이 커집니다.

## 검토 라운드

### 1차 검토

- 검토 내용: 현재 `AGENTS.md`에서 “반드시 상위에 있어야 하는 것”과 “skills/hook로 내려도 되는 것”을 분리했습니다.
- 선택/기각 이유: 핵심 규칙, 최소 작업 흐름, skill/hook 라우팅, 권한/품질의 상위 요약은 유지 대상으로 선택했고, 상세 커맨드 정책과 세부 리뷰 절차는 이관 대상으로 선택했습니다.

### 2차 검토

- 검토 내용: 세부 정책의 수용처를 정했습니다.
- 선택/기각 이유: PR/커밋 예산은 `pr-governance`, PR 생성 방식은 `pr-creator`, 승인 경계는 `command-permission-governance`, 세션/종료 context는 별도 skill로 두는 방향을 선택했습니다.

### 3차 검토

- 검토 내용: 새 구조가 Codex에게 실제로 더 읽기 쉬운지 검토했습니다.
- 선택/기각 이유: `AGENTS.md`를 짧은 인덱스형으로 재작성하고, 세부 동작은 skill 제목으로 찾게 하는 구조를 채택했습니다. 반대로 기존 문서에 링크만 덧붙이는 안은 체감 개선이 약해 기각했습니다.

## 종합 판단

- 최종 선택안: `AGENTS.md`는 상위 계약서/라우터로 축약하고, 상세 절차는 기존 skill과 새 skill로 분리합니다.
- 이유: Codex가 세션마다 빠르게 방향을 잡으면서도, 필요한 세부 규칙은 하위 문서에서 정확히 찾을 수 있기 때문입니다.
- 남은 리스크: 너무 과도하게 축약하면 상위 파일만 보고 바로 행동할 때 규칙이 누락될 수 있으므로, 상위 규칙과 필수 skill 맵은 남겨둡니다.

## 병렬 스트림

- 스트림 A: `AGENTS.md` 재작성 및 상위 라우팅 구조 정리
- 스트림 B: 기존 skill 보강 + 종료 context 전용 skill 추가

## 커밋 예산

- 커밋 1: `AGENTS.md` 축약 재구성 + 계획서 추가, 목표 280~330줄
- 커밋 2: 관련 skill 보강 및 새 skill 추가, 목표 180~260줄

## 수용 기준

- 기준 1: `AGENTS.md`가 핵심 규칙, 최소 플로우, skill/hook 라우팅 중심으로 재구성된다.
- 기준 2: 상세 동작은 skill 또는 hook가 authoritative source로 역할을 가진다.
- 기준 3: 프롬프트 전문가/컨텍스트 엔지니어 관점에서 상위 지시서의 밀도와 라우팅성이 개선된다.
