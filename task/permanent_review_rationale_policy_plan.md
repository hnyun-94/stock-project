# 반복 지침 상시 규칙화 계획서

## 요청 요약

- 모든 답변, 작업, 코드 작성은 다양한 관점에서 여러 번 검토한 뒤 진행한다.
- 선택한 방향에는 명확한 근거와 이유가 있어야 하며, 필요 시 이를 문서나 답변에서 설명할 수 있어야 한다.
- 이 규칙은 일회성 대응이 아니라 프로젝트의 상시 규칙으로 유지되어야 한다.

## 역할별 검토

### PM

- 판단: 타당
- 근거:
  - 반복적으로 같은 요청이 나오고 있어 개인 선호가 아니라 운영 원칙으로 봐야 한다.
  - 품질 기대치가 "빠른 구현"보다 "근거 있는 결정"에 더 가깝다.

### TPM

- 판단: 자동 검증을 일부 도입해야 함
- 근거:
  - 사람 지침만으로는 세션이 바뀌면 누락될 수 있다.
  - 문서, 스킬, pre-push 검증을 같이 걸어야 지속성이 생긴다.

### 기획자

- 판단: 출력 형식을 구조화해야 함
- 근거:
  - "여러 번 검토"는 추상적이라서 최소 산출물 구조가 필요하다.
  - 역할별 검토, 근거, 대안/기각 사유, 최종 판단이 있어야 일관성이 생긴다.

### 개발자

- 판단: 강한 자동화와 과한 규칙의 균형이 필요함
- 근거:
  - 완전 자동 판별은 어렵지만, task/done 문서에 역할 리뷰와 근거가 있는지 정도는 점검 가능하다.
  - false positive를 줄이려면 검사 기준은 작고 명확해야 한다.

### 신입개발자

- 판단: 템플릿 제공이 필요함
- 근거:
  - 규칙만 있으면 실제 작성자가 어떤 형식으로 남겨야 하는지 헷갈릴 수 있다.
  - 재사용 가능한 템플릿이 있으면 품질 편차를 줄일 수 있다.

### 운영자

- 판단: 최종 응답과 문서에 잔여 리스크도 남겨야 함
- 근거:
  - 근거만 있고 남은 리스크가 없으면 운영 판단에 필요한 정보가 빠진다.
  - 사후 회고나 장애 대응에도 같은 구조가 재사용 가능하다.

## 검토 라운드

### 1차 검토: 지침 반영 위치 선정

- 후보 A: `AGENTS.md`만 수정
- 후보 B: `AGENTS.md` + skill 수정
- 후보 C: `AGENTS.md` + skill + template + hook/script

판단:

- 후보 C 채택

이유:

- 문서만 바꾸면 세션 간 일관성이 약하다.
- skill만 바꾸면 사람 눈에 보이는 최상위 규칙이 부족하다.
- hook/script를 같이 두면 최소한의 회귀 방지가 가능하다.

### 2차 검토: 자동 검증 강도

- 후보 A: 경고만 출력
- 후보 B: 코드 변경 시 증빙 문서가 없으면 실패

판단:

- 후보 B 채택

이유:

- 사용자는 "항상 적용"을 명시했다.
- 현재 프로젝트는 이미 pre-push에서 문맥/위생/테스트 게이트를 운영하고 있어 같은 레벨의 강제 정책으로 들어가도 어색하지 않다.

### 3차 검토: 증빙 형식

- 후보 A: 자유 형식
- 후보 B: 역할명 + 근거/이유 + 판단이 들어간 템플릿 기반

판단:

- 후보 B 채택

이유:

- 자유 형식은 자동 검증이 거의 불가능하다.
- 최소 키워드 기반 템플릿은 사람과 도구가 동시에 읽기 쉽다.

## 종합 판단

- 이 지침은 프로젝트 상시 규칙으로 승격하는 것이 타당하다.
- 구현은 `AGENTS.md`, `.agents/skills/review-driven-execution/SKILL.md`, `scripts/check_review_policy.sh`, `.githooks/pre-push`, `scripts/run_quality_gate.sh`, `scripts/README.md`, `task/review_rationale_template.md`를 묶어서 반영한다.

## 병렬 스트림

- 스트림 A: 정책 문서 업데이트 (`AGENTS.md`, 계획서, 템플릿)
- 스트림 B: 실행 가이드 업데이트 (`review-driven-execution` skill)
- 스트림 C: 자동 검증 추가 (`check_review_policy.sh`, pre-push, quality gate)

## 수용 기준

- 상시 규칙이 `AGENTS.md`에 명시된다.
- 실행 스킬이 다중 검토 라운드와 근거 기록을 요구한다.
- 코드/스크립트/워크플로우/스킬 변경 시 증빙 문서 부재를 pre-push에서 차단한다.
- 검증 스크립트와 표준 품질 게이트가 정상 통과한다.
