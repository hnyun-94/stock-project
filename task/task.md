# 주식 리포트 자동화 프로젝트 - Phase 2 고도화 Task 명세서

현재 구축된 MVP 기조를 바탕으로, 프로젝트를 진정한 B2C/엔터프라이즈 레벨로 끌어올리기 위한 **Phase 2 고도화 계획**입니다.
`todo/` 경로의 산재한 요구사항들을 **기획자(PM), 개발자(Engineer), 검토자(QA)** 3가지 역할의 관점으로 구체화하고, 이를 실제 구현하기 위한 진행 Task 목록으로 통합 정리했습니다.

---

## 1. 기획자 (Product Manager) : 플랫폼 매체 확장 및 초개인화 경험

> **목표:** "더 다양한 정보 수집으로 편향성을 없애고, 사용자 개개인의 자산 시나리오에 직결되는 '초개인화' 알림을 제공한다."

### 📝 구체화 내용

- **정보 다각화**: 현재 네이버 뉴스/트렌드에 치중되어 있는 현상을 벗어나, Daum 뉴스, 구글 뉴스 피드, 네이버 데이터랩 검색어를 추가하여 데이터의 커버리지를 늘려야 합니다.
- **포트폴리오 맞춤형(Hyper-Personalization)**: 단순 시장 시황 요약을 넘어서야 합니다. 구독자의 Notion DB에 있는 `보유종목(Holdings)` 데이터를 활용, 오늘 일어난 거시적 사건(금리 인하, 테마주 폭등 등)이 **자신의 포트폴리오에 미치는 영향 및 대응 방안**까지 AI가 제시해야 합니다.
- **이벤트 기반 긴급 알림**: 코스피 대폭락(3% 이상 하락 등)과 같은 비상 상황 발생 시, 지정된 3시간 스케줄을 무시하고 즉각 텔레그램 속보를 푸시(Push)하는 "이벤트 기반 숏 리포팅" 기능의 기획이 필요합니다.

### ✅ 진행되어야 할 Task 목록

- [x] **Task 1.1: 기사/트렌드 데이터 소스 추가 연동**
  - `src/crawlers/daum_news.py`, `google_news.py` 추가 로직 개발
  - 네이버 개발자센터 API 기반 `naver_datalab.py` 연동 및 병렬 파이프라인 병합
- [x] **Task 1.2: 사용자 포트폴리오 스키마 적용**
  - Notion "구독자 관리 DB"에 `보유종목` 및 `긴급알림 임계치` 파라미터(컬럼) 추가
  - `user_manager.py`의 사용자 파싱 과정에 해당 컬럼 수집 데이터 DTO 추가
- [x] **Task 1.3: 초개인화 AI 프롬프트 체인 고도화**
  - Gemini 요약 요청 시 "사용자의 현재 보유종목" 콘텍스트를 강제적으로 주입시켜 개인화 피드백 도달률 극대화

## 2. 개발자 (Software Engineer) : 아키텍처 다변화 및 파이프라인 내결함성(Resilience) 확보

> **목표:** "유지보수하기 좋은 모듈형 구조를 만들고, 트래픽 폭주나 외부 커뮤니티의 봇 차단에도 파이프라인이 멈추지 않게 설계한다."

### 📝 구체화 내용

- **CSR(동적 렌더링) 웹 스크래핑**: 토스 증권 커뮤니티나 블라인드 등 로그인/자바스크립트 렌더링이 필수인 커뮤니티는 현재의 `aiohttp` 요청만으로 파싱이 불가하므로 **Headless Browser(Playwright 등)** 파이프라인 설계 및 격리 수집이 필요합니다.
- **크롤러 레이트 리미트/안티봇 우회**: API 단위 차단(HTTP 429/403) 발생 시, 현재는 즉시 예외가 던져져 전체 병렬 파이프라인(`asyncio.gather`)이 붕괴될 위험이 있습니다. Gemini 호출에만 존재하는 `tenacity` 재시도(Retry/Backoff) 로직을 모든 크롤링 함수에 공통 데코레이터로 이식해야 합니다.
- **AI 프롬프트 추상화/설정화**: 로직 내부 파일(`ai_summarizer.py`)에 하드코딩된 프롬프트 문자열들을 제거하고, 설정 파일(JSON/Markdown)이나 Notion의 별도 Prompt Table에서 텍스트를 읽어오는(Module Decoupling) 구조가 시급합니다.

### ✅ 진행되어야 할 Task 목록

- [x] **Task 2.1: 동적 웹 크롤링 파이프라인 분리 구축**
  - `Playwright` 의존성 설치 및 백그라운드 브라우징 담당자(`dynamic_community.py`) 작성
- [x] **Task 2.2: 네트워크 I/O 재시도(Backoff) 데코레이터 보편화**
  - 레딧, 뉴스 등 모든 외부 HTTP 수집 함수에 `@retry(wait=wait_exponential, stop=stop_after_attempt(3))` 등을 통한 우회 체계 이식
- [x] **Task 2.3: AI Prompt 모듈 및 외부 설정 파일 맵핑**
  - 핵심 프롬프트 지시문(System Prompt)을 `prompts/` 폴더 내 마크다운으로 분리하여 동적 로딩화 구현

## 3. 검토자 (QA / Code Reviewer) : 격리된 테스트 환경 구축 및 사이드 이펙트 방어망 구축

> **목표:** "무한 루프나 네트워크 타임아웃에 의한 스케줄러 먹통을 차단하고, 외부 상태와 무관하게 보장되는 100% 독립 단위테스트를 달성한다."

### 📝 구체화 내용

- **테스트 고립 (Isolation)**: 현재 Github Actions 빌드 검증을 넘기려면 매번 레딧(WSB), Notion DB, Gemini 플랫폼과 실제 통신이 이뤄져야 합니다. 이들 네트워크 장애 시 멀쩡한 우리 코드도 빌드 실패 처리됩니다. **Mocking(모의 객체) 기법**을 전면 도입해 외부 연동 없이 멱등성 있는 CI/CD 환경을 구축해야 합니다.
- **무한 로딩(Hang) 현상 사전 차단**: 크롤링 중 해당 타겟 서버가 응답하지 않는 상황(데드락)에 빠질 경우 스크립트가 영영 멈춰버려 좀비 프로세스가 될 리스크가 존재합니다. 잠재적 행(Hang) 버그 예방을 위해 전체 I/O 코드의 Timeout(타임아웃) 방어 코드가 있는지 전수 리뷰 및 수정이 필요합니다.

### ✅ 진행되어야 할 Task 목록

- [x] **Task 3.1: 모의 객체(Mock API) 단위 테스트 분리 및 작성**
  - `pytest-mock`을 활용해 Network Response를 가짜 데이터(Dummy)로 치환해주는 완전 격리 테스트(`tests/mocked/`) 환경 셋업
- [x] **Task 3.2: 글로벌 Timeout 하드 리미트 강제성 검증**
  - 모든 네트워크 I/O 패키지 (aiohttp 세션 구성, Notion SDK Client, Gemini Connection 등)에 대해 최대 대기 시간(예: Timeout=30s)을 일관되게 명시하도록 코드 점검 및 리팩토링
