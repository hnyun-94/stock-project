# 외부 커넥터 운영 알림 자동화 계획서

## 요청 배경

- 현재 저장소 기준으로 반영 누락된 미완료 장애 수정은 확인되지 않았다.
- Phase 7 백로그에서 가장 우선순위가 높은 미완료 항목은 `외부 커넥터 성공률/지연 임계치 기반 자동 알림(텔레그램)`이다.
- 이미 `external_connector_runs` 테이블에 실행 이력은 저장되고 있으나, 운영자가 바로 행동할 수 있는 경보 계층은 없다.

## 목표

- 외부 커넥터별 1시간/24시간 실패율과 평균 지연을 계산한다.
- 임계치를 넘는 경우 관리자 텔레그램 채널로 1회성 운영 알림을 전송한다.
- 같은 원인으로 알림이 반복 발송되지 않도록 DB 기반 쿨다운을 둔다.
- 알림 실패가 리포트 파이프라인 전체 실패로 번지지 않도록 격리한다.

## 역할별 검토

### PM

- 판단: 채택
- 근거: 사용자는 리포트 품질 저하보다 먼저 데이터 수집 실패를 알아야 한다. 운영 알림은 사용자 가치 손실을 줄이는 직접 수단이다.

### TPM

- 판단: 채택
- 근거: 현재 텔레메트리는 수집만 되고 운영 피드백 루프가 없다. 1h/24h 집계와 쿨다운을 붙이면 관측 가능성과 대응성이 동시에 올라간다.

### 기획자

- 판단: 채택
- 근거: 알림은 "무슨 소스가, 어떤 기준을 넘겨, 왜 위험한지"가 한 번에 보여야 한다. 단순 에러 문자열 전송은 운영 판단에 부족하다.

### 개발자

- 판단: 채택
- 근거: 기존 DB/텔레그램 경로를 확장하는 편이 가장 작은 변경으로 구현 가능하다. 새 인프라 없이도 배치 런타임 안에서 충분히 처리할 수 있다.

### 신입개발자

- 판단: 채택
- 근거: `collect -> persist -> evaluate -> notify` 흐름이 명확하고 테스트 가능하다. 쿨다운 상태를 DB에 두면 재현도 쉽다.

### 운영자

- 판단: 채택
- 근거: 장애 초기에 한 번만 알림을 받고, 정상 구간에 스팸이 없어야 실제 운영에 쓸 수 있다. 최근 오류 사유와 지연 수치가 같이 와야 대응 우선순위를 정하기 쉽다.

## 검토 라운드

### 1차 검토

- 선택안 A: 기존 `send_error_webhook`만 재사용해 예외가 날 때마다 관리자 알림 전송
- 선택안 B: 커넥터 실행 이력 집계 기반 운영 알림 계층을 별도 구현
- 판단: B 채택
- 기각 이유:
  - A는 예외 발생 시점만 포착하고, 반복 장애/성공률 저하/느린 응답을 계량적으로 다루지 못한다.

### 2차 검토

- 검토 내용: 상태 저장 위치
- 판단: 쿨다운과 발송 이력은 SQLite에 저장
- 기각 이유:
  - 메모리 상태는 GitHub Actions 런 간 지속되지 않아 운영 알림 중복 방지에 실패한다.

### 3차 검토

- 검토 내용: 알림 전송 경로
- 판단: `TelegramSender`에 raw `chat_id` 전송 경로를 추가하고, 관리자 chat id 목록을 환경변수로 받는다.
- 기각 이유:
  - 가짜 `User` 객체를 만들어 재사용하는 방식은 목적이 불분명하고 테스트 가독성이 떨어진다.

## 구현 범위

1. `src/utils/database.py`
- 운영 알림 발송 이력 테이블 추가
- 1h/24h 윈도우 기준 source health 집계 메서드 추가
- 쿨다운 확인/발송 기록 메서드 추가

2. `src/services/connector_alerts.py`
- 소스별 실패율/지연 임계치 평가
- 관리자 텔레그램 알림 메시지 생성
- DB 쿨다운 기반 중복 방지

3. `src/services/notifier/telegram.py`
- 관리자용 raw `chat_id` 발송 메서드 추가

4. `src/main.py`
- 외부 커넥터 스냅샷 수집 직후 운영 알림 평가 호출
- 운영 알림 오류는 로깅만 하고 파이프라인은 계속 진행

5. 테스트
- DB 집계/쿨다운 테스트
- 운영 알림 임계치/중복 방지 테스트
- 텔레그램 raw `chat_id` 발송 테스트

## 커밋 예산

1. 계획서 + DB/테스트 뼈대: 140 lines 이하
2. 운영 알림 서비스 + notifier 확장: 260 lines 이하
3. main 연결 + 테스트 보강 + 문서 동기화: 220 lines 이하

## 완료 기준

- 1시간 또는 24시간 윈도우에서 failure rate/avg latency 임계치 초과 시 관리자 알림이 발송된다.
- 같은 fingerprint 알림은 쿨다운 동안 재전송되지 않는다.
- 알림 비활성 또는 관리자 chat id 미설정 시 파이프라인은 정상 진행한다.
- `uv run python -m pytest tests/services/ tests/test_e2e_dryrun.py -q` 통과
- `pre-push` 품질 게이트와 커밋 크기 정책 통과

## 종합 판단

- 현재 가장 가치 높은 미완료 작업은 운영 알림 자동화다.
- 기존 텔레메트리 저장 구조를 재사용하면 구현 비용이 낮고, 운영 가치는 즉시 발생한다.
- 따라서 본 작업을 다음 우선순위로 미루지 않고 바로 구현한다.
