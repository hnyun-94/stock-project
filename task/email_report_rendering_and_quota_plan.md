# 이메일 리포트 렌더링 및 Gemini quota 방어 계획서

## 요청 요약

- 실제 발송된 `.eml`을 기준으로 이메일 리포트의 구성 붕괴와 가독성 저하를 수정합니다.
- 429 `RESOURCE_EXHAUSTED`가 반복되는 구조를 분석하고, retry 중심이 아니라 예산 통제와 호출 재사용 중심으로 방어합니다.
- 앞으로는 기존 역할 리뷰에 더해 `UI/UX 전문가`, `퍼블리셔 전문가` 관점도 상시 포함합니다.

## 역할별 검토

### PM

- 판단: 이번 문제는 기능 부족보다 전달 품질과 운영 구조의 문제라 우선순위가 높습니다.
- 근거: 실제 메일을 받는 사용자는 “데이터가 있느냐”보다 “읽히느냐”와 “실패 없이 오느냐”를 먼저 체감합니다.

### TPM

- 판단: 렌더링 문제와 quota 문제를 별개가 아니라 같은 운영 흐름에서 같이 처리해야 합니다.
- 근거: free-tier 20건/일과 3시간 주기 스케줄은 구조적으로 충돌하며, quota 고갈 시 fallback 품질이 곧 메일 품질 저하로 이어집니다.

### 기획자

- 판단: 이메일은 웹 페이지보다 스캔성이 더 중요하므로 상단 결론, 중단 판단, 하단 상세의 깊이 차를 더 크게 줘야 합니다.
- 근거: 실제 `.eml`에서는 모든 섹션 밀도가 비슷해 어디부터 읽어야 하는지 잘 드러나지 않습니다.

### 시니어 개발자

- 판단: 리포트 builder의 데이터 정제와 formatter의 이메일 전용 렌더링을 동시에 손봐야 합니다.
- 근거: 현재 문제는 CSS만으로 해결되지 않고, 저신호 뉴스 선택과 긴 문장 생성도 함께 얽혀 있습니다.

### 개발자

- 판단: `report_builder.py`, `report_formatter.py`, `ai_summarizer.py`, `main.py`, `database.py`가 함께 수정 대상입니다.
- 근거: 뉴스 정제, HTML 인라인 스타일링, AI 예산 관리, 동일 컨텍스트 재사용이 각각 다른 레이어에 걸쳐 있습니다.

### 품질 담당자

- 판단: 이번에는 “사람이 읽기 쉬운 구조”와 “quota guard” 둘 다 테스트로 고정해야 합니다.
- 근거: 단순 키 존재 테스트로는 저신호 제목 유입, holding watch point 오염, run budget 초과 같은 회귀를 막기 어렵습니다.

### 신입개발자

- 판단: 역할이 다른 섹션은 코드 구조에서도 분리돼야 이해하기 쉽습니다.
- 근거: 이메일 렌더링 전용 함수와 뉴스 relevance helper가 분리되면 왜 이런 규칙이 필요한지 읽기 수월합니다.

### 운영자

- 판단: quota 고갈은 에러를 줄이는 것보다 “오늘은 어디까지 AI를 쓰고 어디서 fallback할지”를 안정적으로 결정해야 합니다.
- 근거: 현재처럼 첫 429 이후 같은 날 다른 런에서도 다시 부딪히면 운영 로그만 누적되고 품질도 불안정해집니다.

### 주식 전문가

- 판단: `Keep에 바로가기`, 소비 뉴스, 포털 잡음이 테마/종목 상단에 올라오는 건 투자 판단을 왜곡합니다.
- 근거: 뉴스 relevance가 약하면 실제로 중요한 수급·실적·매크로 이슈보다 비본질적 제목이 리포트의 첫 인상을 차지합니다.

### 프롬프트 전문가

- 판단: quota가 걸릴수록 규칙 기반 fallback의 문장 구조가 더 중요해집니다.
- 근거: AI 호출이 제한될 때는 fallback 문장 자체가 최종 사용자 경험이 되므로 짧고 구조화된 출력이어야 합니다.

### 컨텍스트 엔지니어

- 판단: 동일 사용자군의 중복 호출과 중복 뉴스 수집은 컨텍스트/호출 비용 둘 다 낭비합니다.
- 근거: 같은 키워드·보유종목 조합이 여러 사용자에게 반복될 때는 런타임 캐시 재사용이 필요합니다.

### UI/UX 전문가

- 판단: 이메일은 헤드라인, 표, 요약 박스가 한눈에 보이도록 시각적 위계가 더 강해야 합니다.
- 근거: 실제 `.eml`은 제목이 일반 문단처럼 보이고, 표와 문단의 대비가 약해 상단 시선 유도가 부족합니다.

### 퍼블리셔 전문가

- 판단: 이메일 클라이언트 호환성을 위해 인라인 스타일과 테이블 기반 래퍼를 더 보수적으로 써야 합니다.
- 근거: head CSS와 div 레이아웃만으로는 일부 메일 클라이언트에서 스타일이 약하게 적용되거나 깨질 수 있습니다.

## 검토 라운드

### 1차 검토

- 검토 내용: 실제 `.eml`과 로그를 기준으로 문제를 `레이아웃`, `콘텐츠 정제`, `quota 구조`로 분해했습니다.
- 선택/기각 이유: “CSS만 수정”하는 안은 기각했고, 내용 선택과 AI 예산 정책까지 포함하는 안을 선택했습니다.

### 2차 검토

- 검토 내용: 429 대응을 retry 개선으로 볼지 budget control로 볼지 검토했습니다.
- 선택/기각 이유: free-tier 구조상 retry 개선만으로는 의미가 작아, `per-run budget + persistent quota block + shared cache` 조합을 채택했습니다.

### 3차 검토

- 검토 내용: 이메일 렌더링을 Markdown 유지형으로 갈지 HTML 전용으로 갈지 검토했습니다.
- 선택/기각 이유: 내부 표준 표현은 Markdown으로 유지하되, 이메일 최종 변환 단계에서 인라인 스타일을 강화하는 하이브리드 방식을 선택했습니다.

## 종합 판단

- 최종 선택안: `콘텐츠 정제 -> Markdown 구조 압축 -> email-safe HTML 인라인화 -> AI 예산 통제 -> 동일 컨텍스트 재사용 -> 리뷰 역할 확장` 순서로 구현합니다.
- 이유: 사용자 체감 문제와 운영 리스크를 가장 적은 변경으로 동시에 줄일 수 있기 때문입니다.
- 남은 리스크: free-tier 자체 한계는 남으므로, 더 많은 AI 요약 품질이 필요하면 결국 상위 요금제 또는 보조 모델 도입이 필요할 수 있습니다.

## 병렬 스트림

- 스트림 A: 뉴스 relevance/리포트 문장 압축/holding watch point 정확도 개선
- 스트림 B: 이메일 HTML 인라인 스타일과 제목/표/섹션 배치 개선
- 스트림 C: Gemini per-run budget, persistent quota block, user 간 재사용 캐시 도입
- 스트림 D: UI/UX 전문가, 퍼블리셔 전문가 역할을 AGENTS/skill/hook/template에 반영

## 커밋 예산

- 커밋 1: 계획서 + AI quota/budget/runtime cache, 목표 260~340줄
- 커밋 2: report builder/formatter/email rendering 개선, 목표 260~360줄
- 커밋 3: 테스트 + 역할 체계 문서/hook/template 동기화, 목표 220~320줄

## 수용 기준

- 기준 1: `.eml` 기준 제목, 요약, 표, 섹션 위계가 메일 클라이언트에서 읽기 쉬운 형태로 나온다.
- 기준 2: `Keep에 바로가기` 같은 저신호 제목이 테마/보유종목 핵심 요약에 노출되지 않는다.
- 기준 3: holding별 watch point가 다른 종목으로 오염되지 않는다.
- 기준 4: Gemini free-tier 조건에서도 같은 날 반복 429를 유발하는 중복 호출이 크게 줄어든다.
- 기준 5: UI/UX 전문가와 퍼블리셔 전문가가 리뷰 정책 문서와 자동 검증 기준에 반영된다.
