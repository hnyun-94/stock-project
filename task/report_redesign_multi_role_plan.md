# 리포트 재설계 및 안전성 강화 멀티롤 계획서

작성일: 2026-03-07

---

## 1. 요청 요약

- 리포트가 과도하게 장황하여 5~10분 내 파악 가능한 길이로 재구성해야 합니다.
- 상단에는 최근 변동과 직전 2개 리포트 대비 주요 변화점을 헤드라인으로 노출해야 합니다.
- 본문은 시간/일/주/월 타임윈도우 슬라이스를 제공하고, 상단일수록 단기 동향, 하단일수록 장기 계획을 담아야 합니다.
- 커뮤니티 데이터는 비속어/민감정보/모델 safety 에러 가능성이 높은 소스를 회피하거나 필터링해야 합니다.
- 포트폴리오 분석은 종합 서술형 대신 보유 종목별 개별 인사이트 중심으로 재편해야 합니다.
- 이번 작업 방식은 앞으로도 반복 가능하도록 문서, 저장 구조, 테스트 구조를 함께 최적화해야 합니다.

---

## 2. 멀티롤 타당성 리뷰

### PM 관점

- 타당성 판단: 매우 높음.
- 이유:
  - 현재 출력은 정보량은 많지만 핵심 행동 포인트가 늦게 등장해 사용자 가치 전달이 약합니다.
  - "헤드라인 + 변화점 + 시간축" 구조는 반복 구독형 리포트 제품과 잘 맞습니다.
- 요구 보완:
  - 섹션별 최대 길이와 우선순위를 명시해야 합니다.
  - 읽기 시간 목표(5~10분)를 테스트 가능한 형식으로 환산해야 합니다.

### TPM 관점

- 타당성 판단: 높음.
- 이유:
  - 요구사항은 프롬프트 변경만으로 해결되지 않고 저장 스키마, 조립 계층, 필터 계층이 함께 필요합니다.
  - "이전 2개 리포트 대비 변화"를 위해서는 리포트 스냅샷 저장이 필수입니다.
- 요구 보완:
  - 새 저장 구조는 DB 마이그레이션 없이 점진적으로 추가되어야 합니다.
  - 필터 실패 시에도 파이프라인이 빈 섹션으로 안전하게 계속 진행해야 합니다.

### 기획자 관점

- 타당성 판단: 매우 높음.
- 이유:
  - 사용자는 "무엇이 바뀌었는지", "지금 뭘 봐야 하는지", "내 종목에 무슨 의미인지"를 원합니다.
  - 리포트를 단기/중기/장기로 내려가는 정보 계층 구조로 재배치하는 것은 UX 관점에서 적절합니다.
- 요구 보완:
  - 헤드라인은 3개 이하 bullet로 제한합니다.
  - 종목별 인사이트는 "상태/근거/액션" 3요소로 고정합니다.

### 개발자 관점

- 타당성 판단: 구현 가능.
- 이유:
  - 현재 `main.py` 중심의 조립 구조를 서비스 레이어로 분리하면 요구를 수용할 수 있습니다.
  - DB, 포매터, AI 요약기, 커뮤니티 크롤러가 이미 나뉘어 있어 확장 지점이 분명합니다.
- 요구 보완:
  - 문자열 기반 섹션 조립을 구조화된 데이터 객체 기반으로 바꿔야 테스트가 쉬워집니다.
  - 안전 필터와 요약 포매터는 AI 호출 전/후에 모두 적용해야 합니다.

### 신입개발자 관점

- 타당성 판단: 높음.
- 이유:
  - 요구사항이 많지만 "입력 정제 -> 구조화 요약 -> 출력 조립"으로 단계화하면 이해 가능합니다.
- 요구 보완:
  - 새 모듈마다 책임을 명확히 문서화해야 합니다.
  - 예시 입력/출력 테스트가 꼭 필요합니다.

### 운영자 관점

- 타당성 판단: 매우 높음.
- 이유:
  - safety 에러를 유발하는 커뮤니티 데이터가 리포트 전체 실패로 번지는 현재 구조는 운영 리스크입니다.
  - 작업 중단 시 이어서 진행할 수 있도록 중간 산출물 저장과 로그 표준화가 필요합니다.
- 요구 보완:
  - 필터링 사유와 사용된 데이터량을 로그/DB에 남겨야 합니다.
  - 커뮤니티 소스 비활성화/제한을 환경변수로 제어할 수 있어야 합니다.

## 3. 종합 판단

- 요청은 제품 가치, 운영 안정성, 유지보수성 측면에서 모두 타당합니다.
- 단순 프롬프트 수정이 아니라 아래 4개 축을 동시에 바꾸는 작업으로 정의합니다.
  - 리포트 정보 구조 재설계
  - 리포트 이력 저장 및 변화점 계산
  - 커뮤니티 safety 방어 로직
  - 종목별 개인화 분석 구조화

개발 착수 결정:
- 착수합니다.
- 단, 한 번의 대규모 문자열 수정이 아니라 병렬 스트림으로 분할하고 테스트를 먼저 보강합니다.

## 4. 리뷰 라운드별 보완 기록

### 라운드 1: 요구 해석 리뷰

- 문제:
  - "이전 2개 리포트 비교"를 기존 데이터만으로는 만들 수 없음.
- 보완:
  - 사용자별 리포트 요약 스냅샷을 DB에 저장하는 신규 경로를 추가합니다.

### 라운드 2: 구현 가능성 리뷰

- 문제:
  - 시장/테마/포트폴리오 텍스트가 모두 자유서술형이라 길이 제한을 강제하기 어려움.
- 보완:
  - 구조화된 섹션 모델과 후처리 요약기, bullet 제한 규칙을 추가합니다.

### 라운드 3: 운영 안정성 리뷰

- 문제:
  - 식갤/종토방 제목은 민감 표현과 비속어가 잦아 safety 에러 및 품질 저하 위험이 큼.
- 보완:
  - 소스 allowlist, 금칙어/패턴 필터, 민감도 점수 기반 제외, 필터 결과 fallback 문구를 추가합니다.

### 라운드 4: 반복 작업 최적화 리뷰

- 문제:
  - 다음 유사 작업에서도 `main.py`가 다시 비대해질 가능성이 큼.
- 보완:
  - 리포트 조립/저장/필터/헤드라인 계산을 별도 서비스로 분리하고 테스트 셋을 마련합니다.

최종 판단:
- 위 보완이 포함되면 요구사항 충족 가능성이 높다고 판단합니다.

## 5. 병렬 실행 스트림

### Stream A: 리포트 구조/저장

- 사용자별 리포트 스냅샷 저장
- 이전 2개 리포트 대비 변화점 계산
- 시간/일/주/월 뷰 데이터 집계

### Stream B: safety/커뮤니티 방어

- 커뮤니티 소스 정책화
- 비속어/민감정보/고위험 패턴 필터
- 필터링 결과 로깅 및 fallback 텍스트

### Stream C: 개인화/포매팅

- 종목별 인사이트 생성 로직
- 상단 단기/하단 장기 구조 포매터
- 읽기 시간/길이 제어 규칙

병합 순서:
- A와 B를 먼저 구현합니다.
- C는 A/B가 만든 구조화 입력을 사용해 최종 조립을 담당합니다.
- `main.py` 접점 수정은 마지막에 한 번만 수행합니다.

## 6. 구현 범위

### 필수 범위

- 사용자별 리포트 요약 스냅샷 영속화
- 최근 3개 리포트 비교 헤드라인
- 시간/일/주/월 타임윈도우 섹션
- 종목별 인사이트 섹션
- 고위험 커뮤니티 필터 및 소스 회피
- 구조화 포매터 + 테스트

### 선택 범위

- 커뮤니티 소스별 품질 점수
- 읽기 시간 추정치 표시
- 장기 전략 섹션에 외부 커넥터 이력 요약 반영

## 7. 수용 기준

- 리포트 상단 첫 화면에서 현재 상황과 주요 변화 3개 이내를 확인할 수 있어야 합니다.
- 전체 리포트는 과도한 장문 단락 없이 bullet/짧은 문단 위주여야 합니다.
- 각 보유 종목마다 최소 1개의 개별 인사이트가 생성되어야 합니다.
- 필터링된 커뮤니티 데이터가 없어도 리포트 생성이 실패하지 않아야 합니다.
- 단위 테스트와 E2E 드라이런이 통과해야 합니다.
