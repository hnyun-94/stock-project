## 리포트 가독성 재설계 계획서

작성일: 2026-03-07
범위: Markdown 구성, 카드 구조, 노이즈 제거, 종목별 차별화, 테스트

### 1. 검토 라운드

#### 라운드 1. 사용자 관점 문제 확인
- 근거:
  - 샘플 리포트가 `짧은 요약 / 근거 1~3 / 긍정 / 중립 / 부정 / 앞으로 예상`을 모든 섹션에 같은 방식으로 반복합니다.
  - `옵션 가이드`, `언론사 선정언론사가 선정한 주요기사...` 같은 포털/스크래핑 노이즈가 그대로 노출됩니다.
  - 삼성전자, SK하이닉스, 엔비디아 섹션이 거의 같은 문장 구조로 반복됩니다.
- 판단:
  - 내용이 없는 것이 아니라, 동일 밀도의 bullet이 과도하게 이어져 시선 우선순위가 무너진 상태입니다.

#### 라운드 2. 기술 구조 확인
- 근거:
  - `src/services/report_builder.py`가 카드마다 동일한 시각 문장을 자동 생성합니다.
  - `src/utils/report_formatter.py`는 모든 정보를 같은 bullet depth로 렌더링합니다.
- 판단:
  - builder는 “무엇을 강조할지”를 더 구조화해야 하고, formatter는 “어떻게 빨리 읽히게 할지”를 더 강하게 책임져야 합니다.

#### 라운드 3. 역할별 리뷰

### PM
- 문제:
  - 사용자가 빠르게 행동 판단을 내려야 하는 리포트인데, 현재는 읽는 데 피로가 큽니다.
- 근거:
  - 핵심 판단과 근거, 시나리오, 체크포인트가 동일한 시각적 무게로 나열됩니다.
- 판단:
  - 각 섹션을 `한줄 판단 -> 왜 그런지 -> 세 가지 시각 -> 다음 체크포인트` 구조로 압축해야 합니다.

### TPM
- 문제:
  - 노이즈 데이터가 formatter까지 전달돼 최종 리포트 품질을 해칩니다.
- 근거:
  - 제목 정제와 포털 chrome 제거가 builder 단계에서 약합니다.
- 판단:
  - title/summary 정제 로직과 저신호 제목 필터를 builder에 추가합니다.

### 기획자
- 문제:
  - 타임윈도우, 테마, 보유종목 섹션이 형식은 다르지만 실제로는 비슷하게 보입니다.
- 근거:
  - 섹션별 의사결정 질문이 다르다는 점이 Markdown 구조에 반영되지 않습니다.
- 판단:
  - 상단은 “오늘 바로 판단”, 중단은 “시간축 판단”, 하단은 “테마/종목/장기 계획” 순으로 더 강하게 분리합니다.

### 개발자
- 문제:
  - 현재 helper가 범용 텍스트를 대량 재사용해 카드별 차별성이 떨어집니다.
- 근거:
  - `_build_context_views()`, `_describe_monitor_points()`가 같은 맥락을 여러 종목에 반복 주입합니다.
- 판단:
  - 종목별/섹션별 watchpoint heuristic과 compact card schema를 추가합니다.

### 신입개발자
- 문제:
  - 지금 구조는 왜 어떤 문장이 summary이고 어떤 문장이 detail인지 파악하기 어렵습니다.
- 근거:
  - formatter가 모든 필드를 동일한 bullet로 출력합니다.
- 판단:
  - formatter를 “섹션 공통 템플릿”으로 재구성해 읽기 규칙이 드러나게 합니다.

### 운영자
- 문제:
  - 리포트 품질이 들쭉날쭉하면 사용자 만족도와 피드백 해석이 왜곡됩니다.
- 근거:
  - 노이즈 기사 제목과 반복 문구는 자동화 리포트에 대한 신뢰를 깎습니다.
- 판단:
  - 노이즈 제거와 시각 구조 개선은 운영 안정성과도 직결되므로 즉시 반영합니다.

### 2. 실행 항목

#### Stream A. Builder 정제
- 저신호/포털 chrome 제목 필터 추가
- 세부 근거 중복 제거
- 종목별 watchpoint 차별화

#### Stream B. Formatter 재설계
- 카드 공통 구조를 Markdown 서브섹션으로 변경
- `세 가지 시각`은 표 또는 compact bullet로 묶기
- `다음 체크포인트`를 별도 시각 구간으로 분리

#### Stream C. 테스트
- noise filter 회귀 테스트
- formatter 출력 구조 회귀 테스트
- 종목별 차별화 테스트

#### Stream D. 문서
- 계획서 및 README의 리포트 구조 설명 반영

### 3. 기각한 대안
- 대안: 텍스트 길이만 줄인다.
- 기각 이유:
  - 길이만 줄이면 구조적 반복과 노이즈 문제가 남아 사용자 피로가 크게 줄지 않습니다.

- 대안: formatter만 바꾼다.
- 기각 이유:
  - builder에서 동일한 문장이 너무 많이 생성돼 있어 formatter만으로는 품질 한계가 있습니다.

### 4. 커밋 예산

- Commit B: formatter 재설계 + 테스트/문서, 목표 320줄 이하

### 근거 또는 이유
- 사용자는 “읽기 쉬운 판단형 리포트”를 원하고 있으며, 현재 문제는 정보량보다 정보 배치와 문장 품질입니다.
- 따라서 리포트 본문은 더 예쁘게가 아니라 더 빨리 이해되게 바뀌어야 합니다.

- builder와 formatter를 함께 손봐야 실제 사용자 체감 품질이 올라갑니다.
