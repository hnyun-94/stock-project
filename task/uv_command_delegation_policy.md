# uv 명령 권한 위임 정책 계획서

## 요청 요약

- 사용자는 앞으로 `uv` 관련 명령어를 자유롭게 사용할 수 있도록 위임하고 싶다고 명시했다.
- 이 지침은 프로젝트 상시 정책으로 유지되어야 한다.

## 역할별 검토

### PM

- 판단: 채택
- 근거: 실제 작업 흐름에서 `uv sync`, `uv run python -m pytest`, `uv run python -m src.main`이 반복적으로 등장하므로 승인 왕복 비용이 크다.

### TPM

- 판단: 조건부 채택
- 근거: 사용자가 명시적으로 위임 의사를 밝혔기 때문에 정책상 반영 가능하다. 다만 `uv` 위임은 광범위하므로, 이 위임이 `uv` 바깥의 고위험 명령까지 확장되지 않는다는 경계는 문서에 남겨야 한다.

### 기획자

- 판단: 채택
- 근거: "uv는 맡기고, 나머지 민감 권한은 별도"라는 규칙이 분명해야 사용자가 승인 피로를 줄이면서도 통제감을 잃지 않는다.

### 개발자

- 판단: 채택
- 근거: 테스트, 런타임 체크, 메인 배치, 피드백 서버, 의존성 동기화까지 모두 `uv` 진입점 하나로 수렴한다. prefix 단위 위임이 가장 실용적이다.

### 신입개발자

- 판단: 채택
- 근거: `uv` 명령만 따로 재확인 규칙에 남겨두면 실제 작업 기준이 자주 흔들린다. 상시 위임으로 고정하는 편이 이해하기 쉽다.

### 운영자

- 판단: 부분 채택
- 근거: `uv run python -m src.main`처럼 실제 외부 발송이 포함된 명령도 사용자 의사에 따라 위임 가능하다. 대신 실행 결과는 항상 명시적으로 보고되어야 한다.

## 검토 라운드

### 1차 검토

- 선택안 A: 기존처럼 `uv` 명령은 일부만 위임하고 실제 발송 런타임은 재확인 유지
- 선택안 B: 사용자의 명시적 요청을 반영해 `uv` prefix 전체를 상시 위임
- 판단: B 채택
- 기각 이유:
  - A는 기존보다 조금만 나아질 뿐이고, 사용자가 명시적으로 위임한 범위를 충분히 반영하지 못한다.

### 2차 검토

- 검토 내용: 위험 경계 표현
- 판단:
  - `uv` 명령은 위임 범위로 이동
  - 다만 `.env`, secret 수정, destructive git, 관리자급 GitHub 명령은 계속 별도 재확인
- 기각 이유:
  - `uv` 위임을 이유로 전반적 권한 완화까지 확장하면 사용자 요청 범위를 넘어선다.

### 3차 검토

- 검토 내용: 실제 반영 위치
- 판단:
  - `AGENTS.md`
  - `.agents/skills/command-permission-governance/SKILL.md`
  - `.codex/config.toml` 주석
  - 본 계획서
  - 가능한 경우 tool prefix approval에 `["uv"]` 저장
- 기각 이유:
  - 문서만 바꾸고 실제 prefix approval을 남기지 않으면 다음 세션의 승인 피로가 완전히 줄지 않는다.

## 종합 판단

- 사용자의 명시적 의사를 근거로 `uv` 관련 명령은 상시 위임 범위로 승격한다.
- `uv run python -m src.main`, `uv run python -m src.apps.feedback_server` 같은 실행형 명령도 재확인 예외에서 제거한다.
- 대신 실행 결과와 영향은 항상 응답에 명시한다.
- `uv` 외의 고위험 명령은 기존처럼 별도 재확인 대상으로 유지한다.
