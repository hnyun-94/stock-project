# 커밋 예산 기반 분할 정책 업데이트

## 요청 요약

- 400줄 이하 커밋 분할은 작업계획서 단계에서 예상 줄수를 계산해 미리 분리하고,
- 실제 커밋 시점에는 매번 수동 검증하지 않아도 되도록 정책을 조정한다.

## 역할별 검토

### PM

- 판단: 채택
- 근거: 사용자의 체감 피로는 커밋 직전 반복 검증보다 계획 단계의 명확한 분할에서 더 크게 줄어든다.

### TPM

- 판단: 부분 채택
- 근거: 계획 단계 예산화는 좋지만, 실제 편집 과정에서 줄수가 불어날 수 있으므로 최종 자동 검증은 남겨야 한다.

### 기획자

- 판단: 채택
- 근거: "계획서에 커밋 예산 명시"는 작업 방식이 더 예측 가능해지고, 이후 회고도 쉬워진다.

### 개발자

- 판단: 부분 채택
- 근거: 수동 `check_commit_size`를 커밋마다 돌리는 것은 비효율적이지만, drift가 생긴 경우를 위한 선택적 수동 검증은 남겨야 한다.

### 신입개발자

- 판단: 채택
- 근거: 언제 수동 검증이 필수인지 헷갈리기 쉬우므로 "기본 생략, drift 의심 시만 수동, 최종은 pre-push" 규칙이 더 이해하기 쉽다.

### 운영자

- 판단: 채택
- 근거: 자동 강제 지점이 `pre-push` 하나로 유지되면 실수 방어는 유지되고 운영 절차는 단순해진다.

## 검토 라운드

### 1차 검토

- 선택안 A: 기존처럼 계획 단계와 커밋 단계 모두 수동 검증 유지
- 선택안 B: 계획 단계에서 예산화하고, 커밋 단계 수동 검증은 기본 생략
- 판단: B 채택
- 기각 이유:
  - A는 안전하지만 반복 비용이 크고, 이미 pre-push 자동 검증이 있어 중복이 많다.

### 2차 검토

- 검토 내용: 자동 검증까지 제거할지 여부
- 판단: 자동 검증은 유지
- 기각 이유:
  - 계획 대비 실제 변경량 drift가 생길 수 있어 최종 안전장치는 필요하다.

### 3차 검토

- 검토 내용: 반영 범위
- 판단: `AGENTS.md`, PR 관련 skill, 계획서 기반 skill, 스크립트 안내 문서까지 동기화
- 기각 이유:
  - 한 곳만 바꾸면 이전 규칙이 다른 문서에 남아 다시 혼선을 만든다.

## 종합 판단

- 앞으로는 계획서 단계에서 커밋 예산을 먼저 잡는다.
- 실제 커밋 시 수동 `check_commit_size`는 기본 필수가 아니다.
- 범위 drift가 의심될 때만 수동 검증을 사용한다.
- 최종 강제 검증은 `pre-push`와 `run_quality_gate.sh`가 계속 담당한다.
