# Gemini Quota Exhaustion 방어 로직 계획서

## 요청 요약

- Gemini `429 RESOURCE_EXHAUSTED`가 발생해 시장 요약, 테마 브리핑, 보유 종목 인사이트가 실패 문자열 또는 지나치게 개략적인 fallback으로 무너집니다.
- 동일 파이프라인 런 안에서 이미 quota 고갈이 확인됐는데도 추가 호출과 재시도가 이어져 품질과 비용 효율이 함께 악화됩니다.
- 목표는 quota 고갈을 조기에 감지하고, 남은 리포트는 구체적인 규칙 기반 fallback으로 유지하는 것입니다.

## 역할별 검토

### PM

- 판단: 즉시 수정 필요
- 근거:
  - 현재 리포트는 핵심 섹션이 `생성 실패` 문구로 노출돼 사용자 가치가 크게 훼손됩니다.
  - 같은 데이터가 있어도 fallback 품질이 낮아 "읽을 만한 5~10분 리포트" 목표를 충족하지 못합니다.

### TPM

- 판단: 런 단위 fail-fast가 필요
- 근거:
  - daily quota 고갈은 같은 런 내에서 회복될 가능성이 낮은데도 재시도와 사용자별 추가 호출이 이어집니다.
  - 쿼터 상태를 공유하지 않으면 batch 실패 후 per-keyword fallback이 다시 API를 두드리는 악순환이 생깁니다.

### 기획자

- 판단: 실패 문구 대신 구체적인 대체 문구가 필요
- 근거:
  - 리포트는 "왜 실패했는지"보다 "지금 무엇을 봐야 하는지"를 전달해야 합니다.
  - fallback이어도 지수/헤드라인/검색 트렌드/종목 뉴스 기반의 구체적 포인트는 제공할 수 있습니다.

### 개발자

- 판단: 서비스 레이어 안에서 quota 판별 + deterministic fallback helper로 푸는 것이 적절
- 근거:
  - `safe_gemini_call()`이 단일 진입점이라 quota 감지와 전역 상태 공유를 넣기 좋습니다.
  - 시장/테마/종목별 로컬 fallback helper를 두면 formatter를 건드리지 않고 품질을 높일 수 있습니다.

### 신입개발자

- 판단: helper를 작게 나눠야 유지보수 가능
- 근거:
  - quota 판별, retry 차단, 시장 fallback, 테마 fallback, 종목 fallback이 한 함수에 섞이면 읽기 어렵습니다.
  - 테스트도 helper 단위로 분리하는 편이 안전합니다.

### 운영자

- 판단: quota 상태 로그를 명확히 남기고 추가 호출을 줄여야 함
- 근거:
  - 장애 원인이 이미 quota 고갈인데 이후 로그가 같은 에러로 도배되면 원인 파악이 더 어려워집니다.
  - "quota 고갈 감지 -> 보수 모드 전환" 로그가 있으면 Actions 장애 판독이 쉬워집니다.

## 검토 라운드

### 1차 검토

- 선택지 A: retry/backoff만 강화
- 선택지 B: 모델 후보를 더 늘려 우회
- 선택지 C: quota 감지 후 fail-fast + 규칙 기반 fallback

- 판단: C 채택
- 선택/기각 이유:
  - A는 실패 문자열과 개략적 리포트 문제를 해결하지 못합니다.
  - B는 쿼터 정책이 모델 단위로 묶이면 부분 완화만 가능하고, free tier 제약 자체는 남습니다.
  - C는 현재 데이터만으로도 품질을 유지하면서 호출 폭주를 끊을 수 있습니다.

### 2차 검토

- 검토 내용: quota 상태를 어디서 공유할지

- 판단: `safe_gemini_call()` 모듈 레벨 상태로 관리
- 선택/기각 이유:
  - 파이프라인 전체에서 Gemini 진입점이 하나라 가장 작은 수정으로 효과를 낼 수 있습니다.
  - DB 저장이나 외부 캐시는 과합니다. 런 단위 메모리 상태면 충분합니다.

### 3차 검토

- 검토 내용: fallback 품질 기준

- 판단: 시장/테마/종목별 helper를 분리하고 모두 구체적인 headline 기반 bullet을 생성
- 선택/기각 이유:
  - 단순 "추가 확인 필요" 문구는 제품 목표를 만족하지 못합니다.
  - 각 섹션이 이미 수집한 원본 데이터로 2~3개의 구체 포인트는 생성 가능합니다.

## 종합 판단

- `safe_gemini_call()`에 quota exhaustion 판별과 런 단위 block 상태를 추가합니다.
- quota 고갈은 재시도 대상에서 제외하고, 이후 호출은 즉시 fail-fast 처리합니다.
- `generate_market_summary()`, `generate_theme_briefing()`, `generate_theme_briefings_batch()`, `generate_holding_insights()`는 모두 구체적인 local fallback을 사용합니다.
- 테스트는 quota 판별, batch quota fallback, 시장 fallback, 종목 fallback 구체성에 초점을 맞춥니다.

## 병렬 스트림

- 스트림 A: quota 감지/차단 로직 구현
- 스트림 B: 시장/테마/보유 종목 fallback 품질 개선
- 스트림 C: 단위 테스트 및 품질 게이트 갱신

## 수용 기준

- `429 RESOURCE_EXHAUSTED` 후 같은 런에서 추가 Gemini 호출이 연쇄적으로 발생하지 않는다.
- 리포트에 `시장 요약 생성 실패`, `테마 브리핑 생성 실패` 같은 실패 문자열이 노출되지 않는다.
- 시장/테마/보유 종목 fallback이 실제 뉴스/지표를 반영한 구체적 문구를 제공한다.
- 관련 서비스 테스트와 전체 품질 게이트가 통과한다.
