# 작업 종료 Context Triage 정책 계획서

## 요청 요약

- 모든 작업이 끝난 뒤 남은 큐를 확인하고, context를 유지할지 초기화할지 판단하는 규칙을 프로젝트에 반영합니다.
- 다만 이 규칙이 실제 context 최적화에 도움이 되지 않거나 오히려 overflow를 키우면 적용하지 않습니다.

## 역할별 검토

### PM

- 판단: 작업 종료 시 context를 정리하는 규칙은 사용자 피로와 반복 설명 비용을 줄이는 데 도움이 됩니다.
- 근거: 후속 작업이 없는데도 긴 과거 맥락을 계속 들고 가면, 다음 요청에서 핵심 판단보다 회고가 더 길어지는 문제가 생깁니다.

### TPM

- 판단: 이 정책은 자동 검증보다 운영 규칙으로 두는 편이 적절합니다.
- 근거: context 가치 판단은 코드 diff처럼 deterministic하지 않아서 hook로 강제하면 오탐과 불필요한 차단이 늘어날 가능성이 큽니다.

### 기획자

- 판단: “큐 확인 -> 유지/압축/초기화 판단 -> 필요한 정보만 보존”의 단순 구조가 가장 이해하기 쉽습니다.
- 근거: 사용자가 원하는 것은 세밀한 메모리 시스템이 아니라, 다음 작업에 도움이 되는 수준의 간결한 문맥 관리입니다.

### 시니어 개발자

- 판단: 이 정책은 문서와 skill에 넣고, 실행 코드는 건드리지 않는 것이 맞습니다.
- 근거: agent context는 저장소 안에서 완전히 제어할 수 없고, 훅으로 검증할 수 있는 성질도 아니므로 behavioral policy가 적합합니다.

### 개발자

- 판단: 구현 변경은 최소화하고 재사용 경로에만 반영합니다.
- 근거: `AGENTS.md`와 종료 흐름을 다루는 skill만 갱신해도 실제 작업 방식에는 충분한 영향이 있습니다.

### 품질 담당자

- 판단: “도움이 될 때만 적용”이라는 예외 조건을 함께 문서화해야 합니다.
- 근거: 좋은 정책도 무조건 적용하면 오히려 불필요한 압축과 정보 손실을 만들 수 있습니다.

### 신입개발자

- 판단: 무엇을 큐로 보고, 무엇을 남겨야 하는지 예시를 적어두는 편이 이해에 유리합니다.
- 근거: context 최적화는 추상적이어서, 미해결 요청/후속 task/실패한 검증 같은 구체 예시가 없으면 실행이 흔들릴 수 있습니다.

### 운영자

- 판단: raw 로그를 오래 붙들기보다 정제 문서 기준선을 우선하는 쪽이 운영 안정성에 유리합니다.
- 근거: 운영 교훈은 `task/`, `done/`, `README.md`, `AGENTS.md`에 정제되어 있어야 다른 세션에서도 일관되게 재사용할 수 있습니다.

### 주식 전문가

- 판단: 도메인 프로젝트에서도 이 정책은 유효하지만, 시장 맥락은 최신성이 중요하므로 오래된 context를 무조건 유지하면 안 됩니다.
- 근거: 주식 리포트 판단은 최근 데이터가 우선이므로, 과거 장황한 설명보다 현재 상태와 검증된 기준선만 남기는 편이 낫습니다.

## 검토 라운드

### 1차 검토

- 검토 내용: 이 규칙을 어디에 넣어야 실제로 쓰이면서도 과도하게 강제되지 않는지 검토했습니다.
- 선택/기각 이유: `AGENTS.md`와 workflow skill 반영을 선택했고, hook 강제는 부적절해 기각했습니다.

### 2차 검토

- 검토 내용: 어떤 정보를 남기고 어떤 정보를 버릴지 정리했습니다.
- 선택/기각 이유: 현재 상태, 핵심 파일, 검증 결과, 잔여 리스크, 다음 액션은 유지 대상으로 선택했고, 긴 raw 로그와 중복 설명은 압축/폐기 대상으로 정했습니다.

### 3차 검토

- 검토 내용: 이 정책의 예외 조건을 검토했습니다.
- 선택/기각 이유: context 최적화에 실익이 없거나 오히려 overflow를 키우면 적용하지 않는다는 조건을 명시하기로 했습니다.

## 종합 판단

- 최종 선택안: Completion Context Triage를 `AGENTS.md`와 종료 흐름 skill에 운영 규칙으로 반영하고, hook에는 넣지 않습니다.
- 이유: 유용한 기준은 되지만 deterministic enforcement 대상으로는 부적합하기 때문입니다.
- 남은 리스크: 실제 context 압축 품질은 작업마다 달라질 수 있으나, 최소한 불필요한 강제 규칙 추가는 피할 수 있습니다.

## 병렬 스트림

- 스트림 B: 종료 흐름과 세션 복원 skill에 정책 반영

## 수용 기준

- 기준 1: `AGENTS.md`에 작업 종료 시 context 유지/압축/초기화 판단 규칙이 명시된다.
- 기준 2: hook 미적용 이유가 문서에 명확히 남는다.
- 기준 3: 관련 skill이 정제 문서 우선/압축 유지 원칙을 반영한다.
